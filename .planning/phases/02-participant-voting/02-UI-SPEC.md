## Spacing Scale

Reused verbatim from Phase 1 — no new tokens. All values are multiples of 4.

| Token | Value | Tailwind class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `gap-1` / `p-1` | Icon-to-label gap inside a grid cell |
| sm | 8px | `gap-2` / `p-2` | Compact spacing (bulk action button row, cell internals) |
| md | 16px | `gap-4` / `p-4` | Default field spacing, grid row gap |
| lg | 24px | `gap-6` / `p-6` | Section padding, card padding |
| xl | 32px | `gap-8` / `p-8` | Major section breaks (form sections) |
| 2xl | 48px | `gap-12` / `p-12` | Page-level vertical padding (`py-12`, matches Phase 1 surfaces) |

**Exceptions (new for Phase 2):**

- `AvailabilityGrid` cell button: minimum height **48px** (`min-h-12`), full row width on mobile (`w-full sm:w-auto`) — exceeds the 44px touch-target floor Phase 1 already established for icon-only controls, because this cell carries icon + text and is the single most-clicked control on the page (click-to-cycle, D-06).
- Bulk-action buttons ("Set all Available" / "Set all Not available" / "Clear"): minimum height 44px (`h-11`), same touch-target rule as Phase 1's date/time inputs.
- Name/email `<input>`: override the shadcn default `h-8` (32px, too small for a touch target) to `h-11` (44px) — same override pattern Phase 1 applied to date/time inputs.

| Role | CSS Variable | Approx. value | Usage |
|------|-------------|----------------|-------|
| Dominant (60%) | `--background` | `oklch(1 0 0)` white | Page background |

**Accent reserved for:**
- The primary submit button on the vote view ("Submit availability")
- The primary save button on the edit view ("Save changes")
- No other element uses `--primary` in Phase 2 (the `AvailabilityGrid` cells use the dedicated vote-state palette below, never `--primary`)

### Vote-state palette (new — additive on top of the base system, same pattern as Phase 1's amber past-date warning)

The three vote states use **light-background + colored border + colored text + a distinct icon** — never a solid saturated fill with white text — for two reasons: (1) it matches the existing badge/warning visual language already shipped (Phase 1 "Keep private" badge: `bg-amber-50 text-amber-700 border-amber-200`), and (2) `-700`-weight text on a `-50` background clears WCAG AA 4.5:1 text contrast comfortably, which a `-500`/`-600` solid fill with white text does not always do.

| State | Value | Icon (lucide) | Label | Tailwind (selected/current cell rendering) |
|-------|-------|----------------|-------|---------------------------------------------|
| Available (`yes`) | Green | `Check` | "Available" | `bg-emerald-50 text-emerald-700 border border-emerald-300` |
| If-need-be (`ifneedbe`) | Amber | `CircleHelp` | "If-need-be" | `bg-amber-50 text-amber-700 border border-amber-300` |
| Not available (`no`) — **default/untouched** | Neutral | `X` | "Not available" | `bg-muted text-muted-foreground border border-border` |

**This is the locked, exact vocabulary** — color is never the sole signal (icon + text label are always rendered together inside every cell, satisfying the existing project a11y precedent of "no color-only communication"). Do not substitute icons or introduce a 4th visual treatment.

No dark mode in Phase 2.

## Component Spec: `AvailabilityGrid`

This is the core net-new component (SPEC VOTE-02, VOTE-07; CONTEXT D-06/07).

### Structure

```
<AvailabilityGrid>
  ┌─ Bulk action row (Label-sized buttons, gap-2, flex-wrap) ────────┐
  │ [Set all Available]  [Set all Not available]  [Clear]            │
  └────────────────────────────────────────────────────────────────┘
  ┌─ One row per candidate date (gap-2 vertical) ─────────────────────┐
  │ {formatDateWithTime label}              [ Icon  State label ]    │  ← single click-to-cycle button
  │ {formatDateWithTime label}              [ Icon  State label ]    │
  │ ...                                                               │
  └────────────────────────────────────────────────────────────────┘
  <input type="hidden" name="votes" value="[{optionId, state}, ...]" />
</AvailabilityGrid>
```

- One row per option, rendered in the SAME chronological order as `getOptionsForPoll` already returns (mirrors Phase 1's date list ordering — no re-sort in the component).
- Date label uses the existing `formatDateWithTime` helper verbatim (timezone-safe, D-11/P3) — same string already used on the participant shell and admin page.
- Each row's interactive element is **one button** (not three) that cycles `no → yes → ifneedbe → no` on each click (D-06 order: Available → If-need-be → Not available → back; default state on first render is `no`).
- **Transparency requirement (prohibition-probe finding):** the default `no` state MUST render with its full visible icon+label ("✕ Not available"), identical in weight to a state the participant explicitly chose — never a blank/dimmed/placeholder-looking cell. A participant who skips a date without clicking it must SEE, before submitting, that it currently reads "Not available" — silence is never visually distinct from an explicit decline, so nothing is submitted that looks like it was merely "left blank."
- The component owns `Record<optionId, VoteState>` client state, serializes it into a single hidden `<input name="votes">` (JSON array of `{optionId, state}`) — the exact pattern `poll-create-form.tsx` already uses for the `dates` hidden input (D-07). The surrounding form posts via `useActionState`.

### Cell anatomy (single button, click-to-cycle)

- Element: `<button type="button">`, full row width on mobile, `min-h-12` (48px)
- Content: `[Icon] [State label text]` — icon is `aria-hidden`, text label is the visible "color is never the only signal" requirement
- `aria-label`: `"{date label}: currently {state label}. Activate to change."` — always describes BOTH the date and the current state, so a screen reader user gets full context from the button alone without needing the adjacent row label
- On click: cycles to the next state, updates the button's visible icon/text/color, AND updates a single page-level `sr-only` `aria-live="polite"` status region with `"{date label} set to {state label}"` — screen readers announce the change even though the button's own accessible name update is not guaranteed to be announced on focus-retention
- Keyboard: native `<button>` — `Enter`/`Space` cycles the state exactly like a click; no custom key handling required; Tab order follows DOM order (one row at a time, top to bottom)
- Visual focus ring: existing shadcn `focus-visible:ring-3 focus-visible:ring-ring/50` (inherited from base `Button`-style classes — do not strip it)

### Bulk actions (VOTE-07)

Three buttons above the grid, `variant="outline"`, `size` default, `h-11` (44px touch target), with icons matching the state vocabulary for visual consistency:

| Button | Icon | Effect |
|--------|------|--------|
| "Set all Available" | `Check` | Sets every rendered option's cell to `yes` |
| "Set all Not available" | `X` | Sets every rendered option's cell to `no` |
| "Clear" | `RotateCcw` | Resets every cell to `no` (same effect as "Set all Not available" — SPEC defines Clear as reset-to-default; both labels are kept because users scan for "Clear" as the reset affordance, mirroring the SPEC's own wording) |

A subsequent single-cell click after any bulk action overrides only that cell (SPEC VOTE-07 AC, D-06) — bulk actions do not lock the row.

**No confirmation dialog on "Clear."** Mirrors the Phase 1 precedent (removing an unsaved date row required no confirmation) — nothing is submitted to the server until the user presses the primary submit/save button, so a misclick is free to undo by re-cycling cells or clicking Clear again.

## Surfaces

### Surface 1: Participant Vote View `/p/[participantUrlId]` — open poll

**Layout:** Centered single-column, max-width 640px (`max-w-2xl mx-auto`), vertical padding 48px — same container as Phase 1's participant shell (this view replaces it).

**Sections (top to bottom):**

1. Poll title (Display / 30px / semibold) — unchanged from Phase 1
2. `PollSummary` (description + location, reused component, unchanged)
3. Section heading: "Your availability" (Heading / 24px / semibold)
4. Name field — required
   - Label: "Your name"
   - Input: `<input type="text" name="name" />`, `h-11`
   - Placeholder: "e.g. Alex"
5. Email field — optional
   - Label: "Email (optional)"
   - Input: `<input type="email" name="email" />`, `h-11`
   - Placeholder: "you@example.com"
   - Helper text (Body / 16px / `text-muted-foreground`): "We'll use this later if the organizer emails a confirmation. No account, no spam."
6. `AvailabilityGrid` (see Component Spec above) — one row per candidate date
7. Submit button — "Submit availability" (variant=`default`, full-width on mobile, auto-width on desktop)

**Same-device preload (D-08):** when the `lfg_edit_<pollId>` cookie resolves to a participant of THIS poll, the page preloads: name field value, email field value, and every grid cell's prior state — **without auto-submitting**. A small inline notice appears above the form: "Showing your previous response. Submit again to update it." (Body / 16px / `text-muted-foreground`). The notice is purely informational — it does not block editing.

### Surface 2: Participant Vote View — closed poll (read-only)

Same layout and sections as Surface 1, with these differences:

- A notice banner replaces the submit affordance area, styled like Phase 1's placeholder banner (`rounded-lg border bg-muted p-6 text-center`):
  - Heading: "Voting is closed" (Heading / 24px / semibold)
  - Body: "The organizer has closed this poll. You can no longer submit or change your availability." (Body / 16px / `text-muted-foreground`)
- Name/email inputs render with the `disabled` attribute (not just visually muted — genuinely non-interactive)
- Every `AvailabilityGrid` cell renders as a non-interactive `<span>` (not a `<button>`) showing the same icon+color+label vocabulary, so a participant who already voted can still SEE their recorded answer
- No submit button is rendered at all (not a disabled button — entirely absent, so there is nothing that looks clickable but silently no-ops)

### Surface 3: `/thanks` Confirmation

**Layout:** Centered single-column, max-width 640px, vertical padding 48px.

**Sections (top to bottom):**

1. Heading: "Thanks for responding!" (Heading / 24px / semibold)
2. Body: "Your availability has been saved." (Body / 16px)
3. Edit-link card (shadcn `Card`, mirrors Phase 1's participant-link card exactly):
   - Label: "Your personal link" (Label / 14px / semibold)
   - Helper: "Bookmark this to change your answer later." (Body / 16px / `text-muted-foreground`)
   - **Privacy warning (prohibition-probe finding, see Prohibitions table):** "Don't share this link — anyone who has it can change your answer." (Body / 16px / `text-amber-700`, same soft-warning treatment Phase 1 used for the past-date badge) — the edit link is an unauthenticated bearer credential (VOTE-06); without this line a participant could forward it alongside the participant link with no idea it lets someone else edit on their behalf
   - URL displayed as truncated monospace text (`font-mono text-sm truncate`): `${base}/p/<participantUrlId>/edit/<editToken>`
   - `CopyLinkButton` (reused component, D-09) with `label="Copy edit link"`
4. Secondary text below the card: "No email was sent — save this link now." (Body / 16px / `text-muted-foreground`) — sets correct expectations since Phase 4 owns email, not this phase

### Surface 4: Edit View `/p/[participantUrlId]/edit/[editToken]`

Same layout/sections as Surface 1 (open poll), with these differences:

- Section heading reads "Edit your availability" instead of "Your availability"
- The grid and name/email fields preload the stored response unconditionally (this route is reached only via the participant's own edit token — no cookie dependency)
- Primary CTA reads "Save changes" instead of "Submit availability"
- On successful save, redirect to `/thanks` (same confirmation surface, same copy — re-saving still shows "Thanks for responding!", not a separate "updated" variant, to keep the copy surface minimal)
- If `poll.status != 'open'`: same read-only treatment as Surface 2 (disabled inputs, non-interactive grid cells, no submit button, "Voting is closed" banner)
- Unknown/invalid `editToken` → `notFound()` (404), same as Phase 1's 404 surface — no new 404 page needed

## Interaction States

### Form — Validation Errors

Same pattern as Phase 1 (`FieldError` component, `role="alert"`, `text-destructive text-sm`, displayed inline below the field).

| Trigger | Field | Error Copy |
|---------|-------|-----------|
| Submit with empty/whitespace-only name | Name input | "Your name is required" |
| Submit with name > 100 chars | Name input | "Name must be 100 characters or fewer" |
| Submit with invalid email format | Email input | "Enter a valid email address" |
| Submit with email > 200 chars | Email input | "Email must be 200 characters or fewer" |

### Form — Submitting / Pending State

Triggered when the Server Action is in-flight (`isPending` from `useActionState`) — same convention as Phase 1's creation form.

- Submit button: disabled, text changes from "Submit availability" → "Submitting..." (vote view) or "Save changes" → "Saving..." (edit view)
- Name/email inputs: `disabled`
- All `AvailabilityGrid` cell buttons and bulk-action buttons: `disabled`

### Wrong/Missing Edit Token (VOTE-06)

Not a form-validation state — handled at the route level via `notFound()` exactly like an unknown `participantUrlId` (Phase 1 precedent). No special "wrong token" copy is shown; the visitor sees the standard 404 surface ("Poll not found" / "This link may be invalid or the poll may no longer exist.") — this avoids leaking whether a token format was merely malformed vs. valid-but-wrong.

### Copy-Link Feedback (`/thanks`)

Identical to Phase 1's admin-page copy behavior — reuses `CopyLinkButton` verbatim, including the "only show 'Copied!' on a resolved clipboard write" prohibition (Phase 1 UI-P2, inherited unchanged, not re-probed here).

## Accessibility Requirements

- All form fields have associated `<label>` elements via `htmlFor`/`id` (shadcn `Label` + `Input`) — same as Phase 1
- Validation error messages use `role="alert"` — same as Phase 1
- `aria-describedby` links each input to its error element when present — same as Phase 1
- **Every `AvailabilityGrid` cell communicates state via icon + text label, never color alone** (locked, see Vote-state palette)
- Every cell button's `aria-label` includes both the date and the current state (`"{date}: currently {state}. Activate to change."`)
- A single page-level `sr-only` `aria-live="polite"` status region announces each cell-state change by date + new state (compensates for the click-to-cycle interaction not natively producing a screen-reader announcement)
- Minimum touch target: 48px for grid cells, 44px for bulk-action buttons and name/email inputs
- Focus order: name → email → bulk-action buttons (in display order) → grid cells (top to bottom, chronological) → submit/save button
- Read-only (closed-poll) surfaces use `disabled` on real inputs and **non-interactive `<span>`s (not disabled buttons)** for grid cells — a disabled button still announces as "button, dimmed" to some AT and can confuse users about whether retrying might work; a plain `<span>` with the same icon+text content avoids that ambiguity
- Cookie-based same-device preload never auto-submits the form — only pre-fills client state; the user must press submit/save for any write to occur (defense against silent overwrite surprises)

## Prohibitions (UI — prohibition-probe)

**Probe policy:** SPEC.md already resolved 5 prohibitions at the requirements level (admin-link leak extends P2, token non-derivation extends P1, status-guard write rejection, email non-leak, token ownership) — these are NOT re-probed here; their server-side verification is owned by SPEC.md / the executor's tests. This section runs the prohibition-probe (recall → precision, per `~/.claude/gsd-core/references/prohibition-probe.md`) over the genuinely NEW UI behaviors this contract adds — the click-to-cycle grid, bulk actions, same-device preload, and the edit-link surfacing — and keeps only the bespoke values/privacy/transparency items; routine engineering candidates (e.g. "Clear must not need a confirm dialog" — already a reversible, pre-submit, low-stakes action per Phase 1 precedent) were dropped at Stage 2.

| Prohibition (must-NOT statement) | Surface | Status | Verification |
|----------------------------------|---------|--------|--------------|
| MUST NOT communicate any vote state using color as the only signal | `AvailabilityGrid` cells (all surfaces) | resolved | test — assert every cell renders both a lucide icon AND a visible text label alongside its color class, for all 3 states |
| MUST NOT render an interactive (clickable/focusable) grid cell, bulk-action button, or submit button when `poll.status != 'open'` | Vote view / edit view, closed-poll state | resolved | test — assert closed-poll render contains no `<button>` for grid cells (only `<span>`), inputs carry `disabled`, and no submit button element exists in the DOM |
| MUST NOT auto-submit the form on same-device cookie preload | Vote view, returning participant | resolved | test — assert that loading the page with a valid preload cookie does NOT trigger a network/server-action call until the user explicitly presses submit |
| MUST NOT surface different 404 copy for "malformed token" vs. "valid-format but wrong/unknown token" | Edit route `/p/[participantUrlId]/edit/[editToken]` | resolved | test — assert both a garbage string and a well-formed-but-nonexistent 21-char token render the identical standard 404 surface |
| MUST NOT present the personal edit link without an explicit "don't share — anyone with this link can change your answer" warning (privacy/transparency; the edit token is an unauthenticated bearer credential, VOTE-06) | `/thanks` edit-link card | resolved | test — assert the `/thanks` page renders the share-warning copy adjacent to the edit-link card, not merely the "bookmark this" helper |
| MUST NOT render an untouched/default grid cell ambiguously (blank, dimmed, or visually distinct from an explicitly-chosen state) such that a participant could submit an unintended "Not available" without seeing that label first (transparency) | `AvailabilityGrid` cells, default state | resolved | test — assert an untouched cell's initial render includes the full "✕ Not available" icon+label, styled identically in weight/opacity to a clicked cell |

**Dismissed (considered, not minted):** a same-device-cookie-miss creating a second/duplicate participant record (e.g. a different browser or cleared cookies) was raised in Stage 1 but is NOT a UI omission — it is a deliberate, already-resolved SPEC/CONTEXT decision (02-SPEC.md Interview Log, Round 2: "no link on a new device = new participant; duplicate names allowed"). No new UI warning is minted for it.

These become `must_haves` for the planner/verifier (negative acceptance criteria), in addition to SPEC.md's own Prohibitions table.
