<objective>
Deliver the participant first-submit slice end-to-end: a participant opens the shared link, marks each candidate date Available / If-need-be / Not available (with VOTE-07 bulk actions), enters a name (optional email), submits, and lands on /thanks with a personal edit link to bookmark. Adds the participants + votes tables, the INSERT-only submitResponse action, the AvailabilityGrid client island, the shared VoteForm, and the live vote view.

Purpose: This is the core differentiator of LFG (VOTE-01/02/03/07). After this plan a brand-new participant can record and persist a three-state response with no account, and walk away with a bookmarkable edit link. The RETURN/edit paths (same-device auto-load + edit link) are 02-02 — they route through updateResponse (D2-05 upsert), so submitResponse here is strictly insert-only (D2-04), mirroring createPoll.
Output: participants/votes schema + local migration, buildEditUrl, submitResponse action (+ DB test), AvailabilityGrid + VoteForm components (+ grid test), the rewritten participant page (submit form / closed-poll read-only), and the /thanks confirmation.
</objective>

## Phase Goal (MVP user story)

**As a** participant in a scheduling poll, **I want to** open the shared link and mark each candidate date Available / If-need-be / Not available without creating an account, **so that** I can tell the organizer when I can meet with zero friction.

This plan is the first vertical slice of Phase 2: after it ships, a participant can complete the full submit-and-confirm journey and receive a bookmarkable edit link. The return-and-edit journey (same-device auto-load + edit link) is the second slice (02-02).

<scope_note>
**Edge-probe resolution (VOTE-01/VOTE-05 idempotency — same-device re-submit):** The deterministic edge-probe asked "what happens if submit runs twice on the same device?" The locked answer (02-SPEC Interview Log Round 2 + D2-05): a same-device re-submit must UPDATE the existing participant, never create a duplicate. That update path is owned by 02-02 (updateResponse upsert + participant-page preload routing). Therefore submitResponse in THIS plan is INSERT-only (D2-04, mirrors createPoll) and the participant page renders a fresh submit form with NO cookie preload yet. Do NOT add same-device preload or insert-or-update branching here — that is deliberately 02-02's slice. 02-01 and 02-02 ship together (prod deploy is the last task of 02-02), so no duplicate-creating state ever reaches users.
</scope_note>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| participant browser -> submitResponse | Untrusted form input (name, email, votes JSON, participantUrlId) crosses into a server action; anyone with the public participant link may submit, by design |
| server action -> Postgres | The action is the sole writer of participants/votes; participant-safe queries are the only readers reaching a participant surface |
| Set-Cookie -> participant device | The httpOnly cookie carries the editToken (a bearer credential) back to the device, consumed by /thanks here and by 02-02's same-device auto-load |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | submitResponse vote rows | mitigate | Server iterates the authoritative getOptionsForPoll list and gap-fills missing options to 'no'; votes for foreign optionIds are ignored (Pitfall 2). Zod enum constrains state to the 3 literals; parameterized db.insert(...).values() is the SQLi defense |
| T-02-02 | Tampering (TOCTOU) | poll.status guard | mitigate | Re-fetch poll.status server-side at write time and reject when != 'open'; never trust a status baked into the rendered form |
| T-02-03 | Information Disclosure | participant view + /thanks payloads | mitigate | getPollByParticipantUrlId selects participant-safe columns only; tests assert no admin_url_id and no '/a/' in the HTML/payload (extends P2) |
