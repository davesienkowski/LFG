# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase adds exactly TWO new visual surfaces to the existing, already-designed
`/a/[adminUrlId]` admin page: (1) a "Who's responded" tracking card, and (2) a
"Nudge non-respondents" control nested inside it. Every token below is the
**existing** project system (detected from `src/app/a/[adminUrlId]/page.tsx`,
`src/components/invite-by-email-form.tsx`, `src/components/results-grid.tsx`) —
nothing here introduces a new palette, type scale, or spacing system. No shadcn
`Badge` primitive is installed in this project; badges are hand-rolled `<span>`
elements matching the established `Booked` / `Best` / `Keep private` idiom.

| Role | Value | Usage |

| Dominant (60%) | `bg-background` (neutral white/near-white) | Page background |

| Accent — emerald (10%) | `border-emerald-300 bg-emerald-100 text-emerald-800` (status badges) / `bg-emerald-50 text-emerald-700 border-emerald-300` (action-result chips) | Success/completion states ONLY |
| Accent — amber (part of the 10%) | `border-amber-200 bg-amber-50 text-amber-700` (status badges) / `border-amber-300 bg-amber-50 text-amber-700` (action-result chips) | Caution/needs-attention states ONLY |
| Destructive | `bg-destructive/10 text-destructive border-destructive/30` | Hard send failures ONLY |

Accent reserved for (explicit list, no new colors added):

- **Emerald** — `Booked` pill, `Best` day badge, invite "Sent" chip, and the **NEW `Responded` badge**. Reserved for confirmed/completed states only.
- **Amber** — `Keep private` admin-link badge, invite "Rate limited" chip, and the **NEW `Not yet responded` badge**. Reserved for pending/needs-attention states — NOT an error state, so it must never share styling with the destructive/red failure chip.
- **Destructive (red)** — invite "Failed" chip and the **NEW nudge "Failed" chip** (reused `SEND_STATUS_META.failed` verbatim). Never used for "not yet responded" — that is an expected, non-error state.
- The "everyone's responded / nudge disabled" affordance copy is **plain `text-muted-foreground`**, not a colored badge — it is a neutral informational state, not a status pill.

## New Surfaces (Phase 7 implementation contract)

### 1. "Who's responded" Card (RSC, admin-only)

**Placement:** New standalone `Card` immediately **below the Results Card**,
**above "Book it"** — same nesting level and shape as the Results Card
(`<Card className="flex flex-col gap-4 p-6">` with an `<h2 className="text-2xl font-semibold leading-snug">` title), so the organizer's flow reads: see the
grid → see who's missing → decide whether to nudge before booking.

**States (mutually exclusive):**

| State | Condition | Render |
|-------|-----------|--------|
| Empty | `invitations.length === 0` | Empty-state body copy only, no badge list, no nudge control |
| Populated, poll open | `invitations.length > 0 && !isClosed` | Summary stat + badge list (`<ul className="flex flex-col gap-2">`, one `<li>` per invitation: email in `font-semibold` + status badge) + nudge control (see below) |
| Populated, poll closed | `invitations.length > 0 && isClosed` | Summary stat + badge list only — **no nudge control at all** (hidden, not disabled — mirrors invite form being hidden when `isClosed`) |

Badge list item markup (mirrors `InviteByEmailForm`'s result list exactly):

```
<li className="flex flex-wrap items-center gap-2 text-sm">
  <span className="font-semibold">{email}</span>
  <span className="inline-flex items-center rounded-full border px-2 py-0.5 text-xs font-semibold ...">
    {Responded | Not yet responded}
  </span>
</li>
```

Responded badge classes: `border-emerald-300 bg-emerald-100 text-emerald-800`.
Not-yet-responded badge classes: `border-amber-200 bg-amber-50 text-amber-700`.
No icon inside these two status badges — matches `Booked`/`Best`/`Keep private`
(text-on-color already satisfies "not color alone" since paired with the text
label, not a bare swatch).

### 2. "Nudge non-respondents" control (client island, mirrors `InviteByEmailForm`)

**Placement:** Nested inside the "Who's responded" Card, below the badge list
(same Card, not a separate one) — button + result-chip list, matching
`InviteByEmailForm`'s internal composition (heading/copy → control → results).

**Gating (all must hold to render the ACTIVE button):**
- `emailConfigured` (same server-only `EMAIL_PROVIDER` check as the page) — if false, hide the control entirely, no explanatory card (the existing "Email isn't set up" fallback Card already covers this once, for invites; do not duplicate it for nudge).
- `!isClosed` — if closed, hide entirely (covered by the Card-state table above).
- `nonRespondentCount > 0` — if zero, render the button **disabled**, replaced/paired with the "Everyone's responded — nothing to nudge." copy (visible, not hidden — a positive completion state the organizer should see).

**Interaction:** `useActionState` + `<form action={formAction}>` with a hidden
`adminUrlId` input, single submit button, `isPending` disables the button and
sets its label to "Nudging…" — identical mechanics to `sendInvites`/
`InviteByEmailForm`. No confirmation dialog (CONTEXT.md: one click = confirmation).

**Server-side recompute (mandatory, prohibition-probe finding — see below):** The
form submits ONLY `adminUrlId`, never a client-rendered recipient list. The
`nudgeNonRespondents` action re-derives the poll from the admin token AND
re-queries current non-respondents from the DB inside the action itself — the
page's server-rendered badge list is a snapshot and MUST NOT be trusted as the
send target. The action also re-checks `poll.status !== "closed"` before
sending, independent of the control being hidden client-side when `isClosed`.

**Results:** `SendInviteResult`-shaped rows (reuse the `sent`/`rate_limited`/
`failed` union and `SEND_STATUS_META` styling verbatim — extract to a shared
module if needed, do not restyle), rendered in an `aria-live="polite"` `<ul>`
that fully replaces itself on each submit, exactly like the invite results list.

## Accessibility Notes (carried convention, not new)

- Status conveyed by icon-or-color is never used alone anywhere in this phase — every badge and chip pairs a text label with its color (WCAG 1.4.1), matching the codebase-wide `SEND_STATUS_META` discipline.
- Nudge result list uses `aria-live="polite"` so the outcome is announced without a focus/navigation change (same as invites).
- Nudge button and any form controls share the `isPending` disabled state — no partial-submit window.
- Card heading is a real `<h2>` at the same landmark level as sibling `Results`/`Book it` headings — no heading-level skip.
- 44px (`h-11`) minimum touch target preserved on the nudge button.

## Prohibition Probe (UI layer)

Manual probe run after this UI-SPEC was drafted: "What could this UI silently
become that the author would NOT want, but the contract does not forbid?"
Scoped to ONLY the two new surfaces this phase adds (does not re-probe anything
already covered/dismissed in 07-CONTEXT.md or the phase SPEC).

| # | Silent-becomes risk | Verdict | Guard now in this contract |
|---|---------------------|---------|------------------------------|
| 1 | Nudge sends to someone who responded moments after the page rendered (stale client-side non-respondent list trusted as the send target) — an unwanted, confusing "why am I still getting reminded?" email | Real — closed | `nudgeNonRespondents` action MUST re-query current non-respondents from the DB server-side at submit time; the form passes only `adminUrlId`, never a recipient list (see "Nudge non-respondents control" section) |
| 2 | Organizer misreads "N of M responded" as total poll participation, when it only reflects people invited through this tool's persisted `invitations` table (RESP-03's documented limitation — direct-link voters are invisible to it) | Real — closed | Mandatory summary caption added: "Only counts people invited by email through this tool." (Copywriting Contract) |
| 3 | Nudge is submitted after the poll was closed/booked in another tab — the control was hidden client-side but the action itself didn't re-check | Real — closed | Action MUST re-check `poll.status !== "closed"` server-side before sending, independent of the client-side hide (see "Nudge non-respondents control" section) |
| 4 | "Responded" badge phrased as a hard identity guarantee ("Confirmed"/"Verified") when it's really just a case-insensitive email-string match (accepted limitation, not solved this phase per CONTEXT.md) | Checked — no action needed | Copy already uses non-absolute wording ("Responded" / "Not yet responded"), never "Confirmed"/"Verified" |

Findings 1–3 are load-bearing for the plan: `gsd-planner` must carry them into
must-haves/acceptance criteria for the nudge server action, not just the UI.
