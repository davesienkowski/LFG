# Phase 8 ŌĆö UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase adds THREE new visual surfaces, all additive to the existing,
already-designed pages: (1) a "Voting deadline" control on `/a/[adminUrlId]`,
(2) a "Your availability" control on `/a/[adminUrlId]` (the organizer's own
row, reusing `AvailabilityGrid` verbatim), and (3) a distinct read-only
closed-copy branch on the participant `VoteForm` (`/p/[participantUrlId]` and
its `/edit/[editToken]` route) for the deadline-passed case. Every token below
is the **existing** project system (detected from
`src/app/a/[adminUrlId]/page.tsx`, `src/components/availability-grid.tsx`,
`src/components/vote-form.tsx`, `src/components/whos-responded-card.tsx`,
`src/components/book-it-control.tsx`) ŌĆö nothing here introduces a new
palette, type scale, spacing system, or component primitive. No shadcn
`Badge`/`Dialog`/date-picker primitive is installed; badges stay hand-rolled
`<span>` elements and the deadline input is a native `<input
type="datetime-local">` wrapped in the existing `Input` component (zero new
dependencies, consistent with the native `<details>` disclosure and native
radio patterns already used on this page).

| Accent ŌĆö amber (part of the 10%) | `border-amber-200 bg-amber-50 text-amber-700` | Caution/needs-attention states ONLY ŌĆö the **NEW** "Voting closed ŌĆö deadline passed" pill/banner, reusing the exact `Keep private` / `Not yet responded` hue |

Accent reserved for (explicit list, no new colors added):

- **Emerald** ŌĆö `Booked` pill (unchanged, `poll.status === "closed"` only),
  and the organizer-availability "Added" confirmation line
  (`text-emerald-700`, matches `bookedLabel`'s
  `text-emerald-700` treatment for a positive-state announcement). Never
  used for the deadline-passed state ŌĆö that is NOT a success state.
- **Amber** ŌĆö the **NEW** "Voting closed ŌĆö deadline passed" pill on the
  admin header/card AND the participant closed-copy heading area is
  **plain text**, not amber ŌĆö amber is reserved for the compact admin-side
  status pill only (mirrors `Keep private`/`Not yet responded`); the
  participant-facing closed banner uses the SAME neutral treatment the
  existing `bookedLabel`-absent branch already uses
  (`text-2xl font-semibold` heading, `text-muted-foreground` body ŌĆö no
  color-coded banner on the participant page, consistent with the existing
  "Voting is closed" copy having no color treatment today).
- **Destructive (red)** ŌĆö reserved for the deadline form's field-level
  validation error only ("Deadline must be in the future"). Never used for
  the deadline-passed *state* itself (that is expected/informational, not
  an error ŌĆö same principle as Phase 7's "not yet responded" vs "failed"
  distinction).
- Mutual exclusivity (load-bearing): the amber "Voting closed ŌĆö deadline
  passed" pill and the emerald "Booked" pill NEVER render together. The
  deadline card (and its pill) is hidden entirely once `isClosed` is true
  (see New Surfaces below) ŌĆö Booked always wins visually once it exists.

## New Surfaces (Phase 8 implementation contract)

### 1. "Voting deadline" Card (client island, admin-only)

**Placement:** New standalone `Card`, same shape as siblings
(`<Card className="flex flex-col gap-3 p-6">`), inserted directly **after
the "poll created" banner and before the candidate-date `<details>`
echo** ŌĆö a poll-level scheduling control belongs at the top of the page,
alongside the title/status area, not buried below the results. Stacking
order becomes: nav ŌåÆ header ŌåÆ justCreated banner ŌåÆ **Voting deadline Card
(NEW)** ŌåÆ candidate-date echo ŌåÆ **Your availability Card (NEW)** ŌåÆ Results
ŌåÆ Who's responded ŌåÆ Book it ŌåÆ Share your poll.

**Visibility:** Rendered ONLY when `!isClosed` (`poll.status !== "closed"`).
Once the poll is booked/finalized, the "Poll finalized" card already
communicates the terminal state; the deadline control is hidden entirely
(not shown read-only) ŌĆö mirrors `InviteByEmailForm`/`showInvite` hiding on
`isClosed`. This is what makes the Booked/deadline-passed pills mutually
exclusive by construction.

**States (mutually exclusive, all rendered when `!isClosed`):**

| State | Condition | Render |
|-------|-----------|--------|
| Unset | `poll.deadline == null` | Heading + "No deadline set ŌĆö voting stays open until you book a date." + datetime input (empty) + "Save deadline" button. No Clear button, no pill. |
| Set, future | `deadline != null && deadline > now` | Heading + "Voting closes {formatted deadline}." + datetime input (prefilled) + "Update deadline" + "Clear deadline" buttons. No pill. |
| Set, passed | `deadline != null && deadline <= now` | Heading + amber pill "Voting closed ŌĆö deadline passed" + "Voting closed on {formatted deadline} ŌĆö the deadline has passed." + datetime input (prefilled, still editable ŌĆö the organizer can push it forward) + "Update deadline" + "Clear deadline" buttons. |

Markup sketch (mirrors `InviteByEmailForm`'s composition: heading/copy ŌåÆ
control ŌåÆ inline result):

```
<Card className="flex flex-col gap-3 p-6">
  <div className="flex flex-col gap-1">
    <div className="flex items-center gap-2">
      <h2 className="text-2xl font-semibold leading-snug">Voting deadline</h2>
      {passed ? <AmberPill>Voting closed ŌĆö deadline passed</AmberPill> : null}
    </div>
    <p className="text-base text-muted-foreground">{state-copy}</p>
  </div>
  <form action={formAction} className="flex flex-wrap items-end gap-2">
    <input type="hidden" name="adminUrlId" value={adminUrlId} />
    <div className="flex flex-col gap-1">
      <Label htmlFor="deadline">Voting deadline</Label>
      <Input id="deadline" name="deadline" type="datetime-local"
             defaultValue={isoLocalOrEmpty} disabled={isPending}
             className="h-11" aria-describedby={error ? errId : undefined}
             aria-invalid={error ? true : undefined} />
      <FieldError id={errId} messages={errors.deadline} />
    </div>
    <Button type="submit" className="h-11" disabled={isPending}>
      {isPending ? "SavingŌĆ”" : hasDeadline ? "Update deadline" : "Save deadline"}
    </Button>
    {hasDeadline ? (
      <Button type="submit" name="intent" value="clear" variant="outline"
              className="h-11" disabled={isPending}>
        Clear deadline
      </Button>
    ) : null}
  </form>
</Card>
```

The datetime input carries a `min` attribute set to the current instant
(client-rendered) as a UX hint only ŌĆö the authoritative "deadline must be in
the future" check happens server-side in the action and surfaces via
`FieldError` (reused from `vote-form.tsx` verbatim), since a `min` attribute
alone is not a security/validation boundary.

**Header pill integration:** the existing header block
(`src/app/a/[adminUrlId]/page.tsx:150-161`) gains ONE new conditional pill,
rendered only when `!isClosed && deadlinePassed`, positioned exactly where
`Booked` would go (same `inline-flex items-center rounded-full ... px-2
py-0.5 text-xs font-semibold` shape, amber palette instead of emerald):

```
{isClosed ? (
  <EmeraldPill>Booked</EmeraldPill>
) : deadlinePassed ? (
  <AmberPill>Voting closed ŌĆö deadline passed</AmberPill>
) : null}
```

### 2. "Your availability" Card (client island, admin-only, organizer's own row)

**Placement:** New standalone `Card`, inserted directly **after the
candidate-date `<details>` echo and before the Results Card** ŌĆö the
organizer adds their own row before reading results so their vote is
already folded into what they see next, matching the natural "set up ŌåÆ
participate ŌåÆ review" flow.

**Reuses:** `AvailabilityGrid` verbatim (same component, same props) for
the per-date three-state input ŌĆö NOT the full `VoteForm` (no email field
needed; name defaults to "You"). A lightweight wrapper form supplies only a
name input + the grid + Save/Cancel.

**Visibility gating (mirrors participant `VoteForm.readOnly` precedent):**

| Condition | Render |
|-----------|--------|
| `isVotingOpen` (poll open, no deadline or deadline not yet passed) | Editable ŌĆö see states below |
| `!isVotingOpen` AND organizer row exists | Read-only summary: heading + "Voting is closed ŌĆö you can no longer change your availability." + `AvailabilityGrid` with `disabled` (the existing closed-chip rendering, reused verbatim) |
| `!isVotingOpen` AND no organizer row yet | Card hidden entirely (nothing to add; mirrors `InviteByEmailForm` hiding on `isClosed`) |

**States (editable branch, `isVotingOpen === true`):**

| State | Condition | Render |
|-------|-----------|--------|
| Empty | no `is_organizer` row yet, form collapsed | Heading "Your availability" + "You haven't added your own availability yet." + **"Add your availability"** button (`type="button"`, reveals the form) |
| Empty, form open | "Add your availability" clicked | Same heading + Name input (`Label` "Display name", `defaultValue="You"`, optional override, `maxLength={100}`) + `AvailabilityGrid` (fresh, all unanswered) + **"Save my availability"** + **"Cancel"** |
| Added, collapsed | `is_organizer` row exists, form collapsed (default) | Heading "Your availability" + emerald confirmation line "Added ŌĆö showing as \"{name}\" in the results below." + **"Edit your availability"** button (outline, reveals the form pre-filled) |
| Added, form open | "Edit your availability" clicked | Same heading + Name input (prefilled with saved name) + `AvailabilityGrid` (`initial` = saved votes) + **"Save my availability"** + **"Cancel"** |

**Interaction:** `useActionState` + `<form>` with a hidden `adminUrlId`
input, identical mechanics to `InviteByEmailForm`/`BookItControl`. "Cancel"
is `type="button"`, collapses the form back to the summary view with zero
side effects and no submit (`BookItControl`'s "Keep poll open" precedent) ŌĆö
it does NOT reset already-saved data, only discards the in-progress edit.
Submit gating mirrors `VoteForm`: the Save button stays disabled until every
candidate date has an explicit state (reuse `allAnswered` logic + the "Choose
an option for every date to submit" helper text verbatim) so the organizer's
row can't be silently saved half-answered either.

**Results-grid integration (presentation only, ORG-01):** the organizer's
row in `ResultsGrid`'s name column appends a muted `" (you)"` suffix when
`is_organizer` is true ŌĆö `<span className="text-muted-foreground">{" "}(you)</span>`
directly after the display name, same row/cell styling as every other
participant otherwise. No badge, no color change, no reordering ŌĆö the row
sorts/renders exactly where any participant row would (best-day computation
and grid layout are untouched per `08-CONTEXT.md`).

### 3. Participant closed-copy branch ŌĆö deadline passed (client, `VoteForm.readOnly`)

**Component:** `src/components/vote-form.tsx`, the existing `readOnly`
sticky-footer block (lines ~180-197). Gains a new `deadlinePassed?: boolean`
prop, computed server-side by both `/p/[participantUrlId]/page.tsx` and
`/p/[participantUrlId]/edit/[editToken]/page.tsx` (never a raw `Date` across
the RSC boundary ŌĆö boolean only, per `08-CONTEXT.md`'s Pitfall-2 precedent).

**Branch logic (replaces the current two-way `bookedLabel ? ŌĆ” : ŌĆ”`):**

```
{bookedLabel ? (
  <>
    <p className="text-lg font-semibold text-emerald-700">
      The group is meeting {bookedLabel}.
    </p>
    <p className="text-base text-muted-foreground">
      The organizer has finalized this date. Voting is closed.
    </p>
  </>
) : deadlinePassed ? (
  <>
    <h2 className="text-2xl font-semibold leading-snug">Voting has closed</h2>
    <p className="text-base text-muted-foreground">
      The voting deadline has passed. Contact the organizer if you need to
      update your availability.
    </p>
  </>
) : (
  <>
    <h2 className="text-2xl font-semibold leading-snug">Voting is closed</h2>
    <p className="text-base text-muted-foreground">
      The organizer has closed this poll. You can no longer submit or
      change your availability.
    </p>
  </>
)}
```

The container itself (`sticky bottom-0 ... bg-muted ...`) is unchanged ŌĆö
only the inner heading/body copy branches three ways instead of two. No new
color treatment: the deadline-passed branch uses the exact same neutral
`text-2xl font-semibold` / `text-muted-foreground` pair as the existing
generic-closed branch (this is a copy change, not a visual-severity change ŌĆö
a passed deadline is informational, not an error).

## Accessibility Notes (carried convention, not new)

- Status conveyed by icon-or-color is never used alone anywhere in this
  phase ŌĆö the "Voting closed ŌĆö deadline passed" pill pairs its amber
  background with an explicit text label (WCAG 1.4.1), matching every other
  hand-rolled pill on this page.
- The datetime input has an associated `<Label htmlFor>` and a field-level
  `FieldError` wired via `aria-describedby`/`aria-invalid`, reusing
  `vote-form.tsx`'s existing `FieldError` component verbatim.
- The organizer-availability form reuses `AvailabilityGrid`'s existing
  radiogroup semantics (one `role="radiogroup"` per date, `aria-live="polite"`
  announcement region, keyboard-reachable `role="radio"` buttons) with ZERO
  changes to that component ŌĆö this phase only supplies new callers.
- All new buttons (Save/Update/Clear/Add/Edit/Cancel) keep the 44px
  (`h-11`) minimum touch target and share their form's `isPending` disabled
  state ŌĆö no partial-submit window, consistent with every existing form on
  this page.
- Both new admin Cards use a real `<h2>` at the same landmark level as
  sibling `Results`/`Who's responded`/`Book it` headings ŌĆö no heading-level
  skip introduced by inserting two new cards into the stack.
- The participant closed-copy branch keeps its heading as a real `<h2>` (not
  a `<p>` masquerading as a heading) in all three cases, preserving the
  existing landmark structure of that sticky footer.

## Prohibition Probe (UI layer)

Manual probe run after this UI-SPEC was drafted: "What could this UI silently
become that the author would NOT want, but the contract does not forbid?"
Scoped to ONLY the three new surfaces this phase adds (does not re-probe
anything already covered/dismissed in `08-CONTEXT.md` or the phase SPEC).

| # | Silent-becomes risk | Verdict | Guard now in this contract |
|---|---------------------|---------|------------------------------|
| 1 | The organizer edits their own availability on a poll that's already booked/finalized, because the control stayed visible and merely looked disabled instead of being removed ŌĆö implying votes still matter post-finalization | Real ŌĆö closed | "Your availability" Card visibility table: hidden entirely (no row yet) or read-only-summary (row exists) whenever `!isVotingOpen`; the editable form (grid + Save) never renders once voting is closed, matching `InviteByEmailForm`'s hide-on-`isClosed` precedent |
| 2 | The amber "Voting closed ŌĆö deadline passed" pill renders alongside the emerald "Booked" pill on a poll that was booked after its deadline passed, producing two conflicting status pills next to the title | Real ŌĆö closed | Header pill integration is an `if/else if` (not two independent conditionals): `isClosed` always wins, `deadlinePassed` pill only evaluated in the `else` branch; the deadline Card itself is hidden entirely once `isClosed` (Surface 1 visibility rule) |
| 3 | A participant whose poll closed due to a passed deadline reads the generic "The organizer has closed this poll" copy and assumes the organizer manually closed it (loses the "just chase the organizer, nothing you did wrong" nuance and may think re-submitting is possible some other way) | Real ŌĆö closed | Surface 3's three-way branch gives deadline-passed its own distinct heading/body ("Voting has closed" / "The voting deadline has passed...") ŌĆö never falls through to the generic organizer-closed copy when `deadlinePassed` is true |
| 4 | The organizer sets a deadline in the past (typo, timezone confusion) and the form silently accepts it, instantly and confusingly closing voting with no feedback about why | Real ŌĆö closed | Dedicated field-level error copy "Deadline must be in the future" (`role="alert"`, `FieldError` pattern) ŌĆö the Save/Update action must reject a non-future deadline server-side, not just via the `min` HTML hint |
| 5 | The organizer's display name literally collides with an existing participant who typed "You", creating two visually identical "(you)"-suffixed-looking rows | Checked ŌĆö no action needed | Cosmetic-only risk; `08-CONTEXT.md` already enforces at-most-one `is_organizer=true` row server-side (data integrity unaffected), and the "(you)" suffix is appended ONLY to the row where `is_organizer` is true, never inferred from the name string ŌĆö a participant named "You" simply has no suffix |

Findings 1ŌĆō4 are load-bearing for the plan: `gsd-planner` must carry them
into must-haves/acceptance criteria for the deadline action, the
organizer-availability action, and the participant page's `deadlinePassed`
prop wiring ŌĆö not just the visual layer.
