<objective>
Replace the repeating "Add date" rows on the creation form (`/`) with a month-calendar multi-select +
a "Selected dates" side list (optional per-date start time) + an "Apply to all" control, per
`01-04-UI-SPEC.md` (Surface 1 delta) and the approved design `01-04-DESIGN-calendar-date-selection.md`.

This is a **Phase 1 revision** satisfying **POLL-05** (and re-satisfying POLL-03/POLL-04 through the new
picker). It is almost entirely a client-component swap: **no schema change, no createPoll/Zod change, no
change to the hidden `dates` submit contract**. The calendar simply produces more of the same
`{date, startTime|null}` rows the action already consumes.

The single load-bearing risk is the **input-layer timezone bug** (third surface of PLAT-04 / P3): a
selected JS `Date` converted via `toISOString().slice(0,10)` shifts to the previous calendar day under
some offsets. The conversion helper builds the string from **local** components and is locked by a
dual-TZ prohibition test.

Output: a working calendar picker on `/`, all existing Phase 1 tests still green, a new input-layer TZ
test green under UTC±extremes, and a production redeploy to Vercel.
</objective>

## Task 2 — Conversion helper + dual-TZ prohibition test (TDD: RED → GREEN)

- **RED:** write `src/lib/date-input.test.ts` first:
  - `toLocalDateString(new Date(2026, 9, 4))` === `"2026-10-04"` (Sat Oct 4) — the assertion holds
    under UTC+14 and UTC-12 because construction + read are both local.
  - month/year rollover: `toLocalDateString(new Date(2026, 0, 1))` === `"2026-01-01"`;
    `toLocalDateString(new Date(2026, 11, 31))` === `"2026-12-31"`.
  - zero-padding: single-digit month/day are padded.
  - `localTodayDateString()` returns a `^\d{4}-\d{2}-\d{2}$` string (diagnostic; no drift assertion).
  - Header comment documents the two-invocation harness (mirror `format-date.test.ts`):
    `TZ=Pacific/Kiritimati npx vitest run src/lib/date-input.test.ts` and `TZ=Etc/GMT+12 ...`.
- **GREEN:** implement `src/lib/date-input.ts`:
  - `toLocalDateString(d: Date): string` — `${d.getFullYear()}-${pad(d.getMonth()+1)}-${pad(d.getDate())}`.
    No `toISOString`, no `Date.UTC`, no UTC getters.
  - `localTodayDateString(): string` — same, off `new Date()` (reuse the date-row `todayLocalIso` idea).
  - File header explains the input-layer footgun (symmetry with `format-date.ts`'s storage/display note).
- Run the test under BOTH TZs; both green.

## Task 3 — Build the calendar picker client component

- Create `src/components/calendar-date-picker.tsx` (`"use client"`):
  - State: `days: Date[]` (the user's clicked Date objects — source of truth for the calendar) +
    `times: Record<string, string>` keyed by canonical `toLocalDateString(day)` + a `defaultTime` string.
  - `<Calendar mode="multiple" selected={days} onSelect={...} disabled={{ before: localMidnightToday }} />`
    where `localMidnightToday = new Date(y, m-1, d)` built from local components (NOT `new Date(isoString)`).
  - `onSelect` (react-day-picker hands back the full new `Date[]`): adopt it as `days`; drop `times`
    entries for removed days; new days default to blank time. Each calendar day is inherently unique →
    no duplicate date possible (P-dupe).
  - Default-start-time `<input type="time">` + "Apply to all" `<button type="button">`: writes
    `defaultTime` into `times` for **every** day in `days` (blank → clears all) (P-applyclear).
  - Selected-dates list: derive `sorted = days.map(d => ({day:d, date: toLocalDateString(d)}))` sorted by
    `date` then blank-time-first; render `formatDateOnly(date)` label (SAME canonical string → single
    source of truth), a per-date `<input type="time">` bound to `times[date]`, and a `×` remove button
    (`aria-label={`Remove ${formatDateOnly(date)}`}`).
  - Empty state copy when `days.length === 0`: "Pick days on the calendar to add candidate dates."
  - Emit upward via `onChange(sortedPayload)` where `sortedPayload = sorted.map(({date}) => ({ date,
    startTime: times[date] || null }))` — chronologically sorted, deduped, P3-safe (P-order).
  - All controls respect a `disabled` prop (pending state).
- Add a component test `src/components/calendar-date-picker.test.tsx` (`// @vitest-environment jsdom`):
  toggle on/off/on → one entry (P-dupe); pick out of order → sorted payload (P-order); apply-to-all
  stamps all, blank clears all (P-applyclear); past day disabled / not selectable (P-pastghost). Use the
  exported `onChange` payload as the assertion surface.
