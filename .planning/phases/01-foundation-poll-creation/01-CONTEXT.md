# Phase 1: Foundation & Poll Creation - Context

**Gathered:** 2026-06-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the foundation (Next.js 16 + Drizzle + Postgres-everywhere + nanoid tokens) and the first vertical slice: an organizer creates a poll (title, optional description/location, ≥1 candidate date with optional time) and lands on an admin page exposing two distinct, unguessable share links. Runs on local Docker Postgres and deploys to Vercel/Neon, both free tier. No voting UI, no results grid, no email — those are later phases.

<decisions>
## Implementation Decisions

### Scaffold & Project Structure
- **D-01:** Bootstrap with `create-next-app` — Next.js 16, App Router, TypeScript, Tailwind CSS, ESLint, `src/` directory. Rationale: STACK.md recommendation; App Router + Server Actions removes a separate API layer for this CRUD scale.
- **D-02:** Drizzle ORM + drizzle-kit. Schema and client live under `src/lib/db/` (`schema.ts`, `index.ts`); server actions under `src/lib/actions/` (e.g. `create-poll.ts`).

### Database Connection Strategy
- **D-03:** One Drizzle schema, two drivers selected by environment: `node-postgres` (`pg`) against a local Docker Postgres in development; Neon serverless (pooled) on Vercel. Rationale: Postgres-everywhere avoids the SQLite/Postgres dialect divergence (PITFALLS.md); Neon chosen over Supabase for ~1–3s resume vs ~30s pause.
- **D-04:** Local dev Postgres via Docker Compose (documented `docker compose up`); connection via `DATABASE_URL`. Neon via `DATABASE_URL` (pooled connection string) on Vercel. Migrations applied with `drizzle-kit`.

### Schema Shape
- **D-05:** `polls` table: internal PK (uuid), `participant_url_id` (text, unique, indexed), `admin_url_id` (text, unique, indexed), `title` (text, not null), `description` (text, nullable), `location` (text, nullable), `status` (text, default `'open'`), `created_at` (timestamptz default now).
- **D-06:** `options` table: PK, `poll_id` FK → polls (cascade delete), `date` (Postgres `DATE`, not null), `start_time` (`TIME`, nullable), `position` (int for stable ordering). Unique constraint on `(poll_id, date, start_time)` to enforce duplicate-date dedupe at the DB level.

### Token Strategy
- **D-07:** `participant_url_id` and `admin_url_id` are each generated by an **independent** `nanoid(21)` call (≥126-bit entropy). The admin token is NEVER derived from the participant token (load-bearing prohibition P1 in SPEC.md). On a unique-constraint collision, regenerate and retry.

### Routing & Pages
- **D-08:** Routes — `/` (creation form), `/a/[adminUrlId]` (admin page showing both links), `/p/[participantUrlId]` (participant shell that resolves the poll; voting UI deferred to Phase 2). Unknown/garbage token → Next.js `notFound()` (HTTP 404), satisfying the non-enumerability acceptance check.
- **D-09:** The participant page and its payloads MUST NOT contain `admin_url_id` (prohibition P2). Queries for the participant route select only participant-safe columns.

### Link Display
- **D-10:** Absolute share URLs built from `NEXT_PUBLIC_BASE_URL` (env), falling back to request headers when unset. Admin page renders both URLs with copy-to-clipboard buttons, clearly labelled "Participant link" and "Admin link (keep private)".

### Date Handling (PLAT-04 / prohibition P3)
- **D-11:** Date-only values are stored as Postgres `DATE` and handled in app code as `'YYYY-MM-DD'` strings. NEVER construct `new Date("YYYY-MM-DD")` (UTC-parse footgun). Rendering/formatting for date-only uses string-based formatting or a date lib configured to avoid timezone shifting. A test renders a date under TZ `Pacific/Kiritimati` (UTC+14) and `Etc/GMT+12` (UTC−12) and asserts identical calendar day.

### Claude's Discretion
- Exact form component breakdown, Tailwind styling specifics, validation library (e.g. zod) vs hand-rolled, and the date-formatting helper are left to the planner/executor, provided D-11's no-`new Date()` rule and the SPEC acceptance criteria hold.

</decisions>
