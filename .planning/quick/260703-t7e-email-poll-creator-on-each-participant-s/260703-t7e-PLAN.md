<objective>
Email the poll CREATOR every time a participant records their availability — on the FIRST submit (submit-response.ts) and on any later EDIT (update-response.ts) — naming the participant and linking the `/a/` admin results view. This requires persisting the creator's email, which quick task `rqc` left transient: add a NULLABLE `polls.creator_email` column (migration 0004), store it in `createPoll`, then fire a SEPARATE best-effort `after()` notification from both participant actions when a stored creator email exists.

Purpose: The creator currently learns nothing when responses arrive — they must keep the admin tab open and refresh. An opt-in per-response notification (naming who responded, linking straight to the live results) closes that loop with no accounts and no cost.

Output: one NULLABLE column + generated migration 0004 (applied LOCALLY only), a persist line in `createPoll`, a new server-side notify-targets query, a new creator-recipient email template, best-effort `after()` wiring in both participant actions, and tests. Email stays FULLY optional (D-02): with no stored creator email OR `EMAIL_PROVIDER` unset, creation/submit/edit behave exactly as today.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| participant browser -> submit/update server action | Untrusted form input (name, email, votes, editToken) crosses here. |
| server action -> creator's inbox | The creator-notify email carries an `/a/` admin URL — a bearer secret that must reach ONLY the creator. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-t7e-01 | Information Disclosure | admin_url_id in participant actions | mitigate | admin_url_id is resolved ONLY via getPollAdminNotifyTargets and used ONLY inside the after() closure to build the creator email; never returned to the page, RSC props, or the participant browser (getPollByParticipantUrlId/getParticipantByEditToken keep omitting it). Explicit code comment + verifier grep enforce this. |
| T-t7e-02 | Information Disclosure | creator_email column | mitigate | creator_email is selected ONLY by getPollAdminNotifyTargets (server-side, inside after()); no participant-facing read (getPollByParticipantUrlId, getResultsForPoll, getFinalizedPollByParticipantUrlId) selects it. |
| T-t7e-03 | Tampering (HTML/markup injection) | participantName in renderParticipantResponseNotification | accept | participantName (user input, max 100 chars, Zod-validated) is interpolated EXACTLY as `title` is across every existing template via renderShell — no new raw-HTML path is introduced. This matches the app's established template-injection posture (T-04-02 templates already interpolate title the same way); email clients render, not execute, and no new escaping regression is created relative to the existing baseline. |
| T-t7e-04 | Denial of Service | best-effort send failure | mitigate | The send is scheduled via after() with its result ignored; a transport failure (or EMAIL_PROVIDER=none) never throws past after() and never affects the participant's redirect (D-02). |
| T-t7e-05 | Spoofing (misdirected recipient) | creator-notify `to` address | mitigate | `to` is the poll's own stored creator_email, read server-side by poll.id — never a client-supplied address; EMAIL_FROM is fixed by the sendEmail seam (D-03). |
| T-t7e-06 | Information Disclosure (over-disclosure, F2) | participant email/edit token in the notification | mitigate | renderParticipantResponseNotification's signature carries NO email/token param and the actions pass participantName: name only — the notification names the participant but can never surface their private email or edit credential to the creator (matches getResultsForPoll/getVoterEmailsForPoll no-email discipline). Enforced by the template signature + an F2 non-vacuous canary test. |
