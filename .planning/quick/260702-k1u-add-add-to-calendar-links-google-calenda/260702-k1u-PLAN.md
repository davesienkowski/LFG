<objective>
Add two "Add to Calendar" mechanisms for the booked (winning) date to the
finalization email sent when the organizer clicks "Book it" (`closePoll`):

1. An **"Add to Google Calendar"** link (plain `calendar.google.com/calendar/render?action=TEMPLATE` URL, no backend), rendered in the email body.
2. A **hosted `.ics` (iCalendar)** file served by a GET Route Handler at `/p/{participantUrlId}/event.ics` (`Content-Type: text/calendar`), rendered as a clickable link in the email AND attached to the finalization email so iOS Mail shows its native "Add to Calendar" banner.

Purpose: recipients of the finalization notice can one-click add the agreed
session to their calendar, on both Google and Apple/Outlook clients.

Output: a pure `src/lib/calendar/links.ts` builder module (+ tests), an
attachment-capable `sendEmail` seam, an extended finalization template, a new
participant-safe query, the hosted `.ics` route, and best-effort wiring in
`closePoll`.
</objective>

<decisions>
- **Timed-event default duration = 3 hours (180 min).** A D&D session is long; a
  1h default undersells it. Applied to BOTH the Google link end time and the
  `.ics` DTEND when `startTime` is set. Called out here per the task spec.
- **All-day event when `startTime` is NULL.** Google `dates=YYYYMMDD/YYYYMMDD+1`
  (end exclusive next day); `.ics` `DTSTART;VALUE=DATE` / `DTEND;VALUE=DATE`
  (next day). No time invented.
- **Floating local time when `startTime` is set** (no TZID, no `Z` on
  DTSTART/DTEND). The app deliberately stores no timezone (D-11/P3); a floating
  time is the correct drift-free representation.
- **`.ics` route keyed by `participantUrlId`** (the token the finalization-email
  recipient already holds via the participant link). The winning date is not
  secret, so no admin token is required; the route only serves a **closed**
  poll. Participant-safe columns only — never `admin_url_id`.
- **No location** in the calendar event (poll `location` is free-text and often
  informal; keep the event minimal — `buildIcs` emits NO `LOCATION` property).
  Title falls back to poll description, then "LFG event", when the poll title is
  empty OR whitespace-only.
- **Event notes = poll description when present, omitted otherwise.** Spec-aligned
  with the Google `details=<desc>` template. The `.ics` `DESCRIPTION` property and
  the Google `details` param are emitted ONLY when the description is non-empty —
  never an empty `DESCRIPTION:` line (CAL-08 edge). Poll description is already
  visible to all participants, so it is not a new disclosure.
</decisions>

<edge_probe_resolutions>
Ran `edge-probe.cjs` over the 10 NEW calendar requirements (CAL-01..10) plus a
manual prohibition-probe. Findings folded below; the rest were dismissed with
source-backed reasons.

FOLDED (see the tasks):
- CAL-04 adjacency / prohibition: all-day `DTEND` is the day AFTER (end-exclusive)
  — MUST be `!= DTSTART` and exactly `+1 day`, INCLUDING month/year/leap rollover
  (a naive string day-increment breaks 2026-12-31; use `Date.UTC` arithmetic).
  Tested with a year-boundary case. Google all-day second date follows the same
  +1-day rule.
- CAL-05 adjacency: timed rollover past midnight — tested at BOTH a non-midnight
  case (22:00 + 3h = next-day 01:00:00) and the exact-midnight boundary
  (21:00 + 3h = next-day 00:00:00).
- CAL-08 empty: OMIT the `.ics` `DESCRIPTION` / Google `details` when the poll
  description is empty/null (never an empty property line). UID is always present
  (poll id + option id).
- CAL-08/CAL-02 encoding: `.ics` text escaped BEFORE line-folding; fold at
  code-point boundaries (spread iteration, not code-unit) so a fold never splits a
  surrogate pair. Route declares `charset=utf-8`; attachment content via
  `Buffer.from(content)` (utf-8 default).
- CAL-10 must-NOT: whitespace-only title ("   ") and empty-description both fall
  through to "LFG event"; `buildIcs` emits no `LOCATION`.
- CAL-06 must-NOT: the `buildGoogleCalendarUrl`/`buildIcs` calls in `closePoll`
  MUST sit inside the try/catch that runs BEFORE `after()` — a synchronous throw
  must not block the redirect nor the scheduled sends; it degrades to no calendar
  block/attachment and the committed close stands.
- prohibition: invite/confirmation emails carry NO attachments (`attachments` is
  OPTIONAL); their existing sends stay byte-identical.

DISMISSED (source-backed):
- CAL-02/03 concurrency: the route is a read-only idempotent GET with no shared
  mutable state; the `.ics` content is built ONCE and reused (immutable) across the
  dedup loop.
- CAL-04/05 ordering: no collection to order; `DTEND > DTSTART` is structural.
- CAL-07 must-NOT: `status !== "closed"` → 404, participant-safe query omits
  `admin_url_id`, unknown and open both return an IDENTICAL 404 (no oracle),
  `Cache-Control: no-store` — all in-plan and tested.
- CAL-09 must-NOT: calendar code lives ONLY in the email/route; the admin-page
  MAIL-03 "Email isn't set up" fallback UI has zero calendar code; the `none`
  branch of `sendEmail` returns before ever touching `attachments`.
- Enumeration/header-injection/floating-as-UTC: keyed by the unguessable
  participantUrlId (same threat model as the participant page), fixed `event.ics`
  filename (no title-derived header), and floating time is the deliberate
  no-timezone representation.
</edge_probe_resolutions>
