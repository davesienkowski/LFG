<objective>
Redesign the admin (`/a/[adminUrlId]`) and participant (`/p/[participantUrlId]`)
page LAYOUTS to use desktop width well and present long candidate-date lists far
more compactly, following the LOCKED "Recommended direction" + "Concrete layout
changes checklist" in 260703-tv3-RESEARCH.md.

Five sequential, presentation-only changes:

1. A condensed, timezone-safe date formatter family (highest-leverage; halves the
   width of every repeated date) — plus a month-year formatter for grouping.
2. Participant surface: widen the shell and turn AvailabilityGrid's desktop
   single-column stack into a responsive 2-column matrix.
3. ResultsGrid: a functional sticky header row (the ONLY component-behavior touch).
4. Admin surface: a two-column dashboard shell + condensed candidate-date echo.
5. BookItControl: a denser candidate-date radio grid.

Purpose: on a poll with 17+ candidate dates, both pages currently cap at a 672px
column with one full-sentence date per line, wasting ~700px of gutter on each
side of a 1440px screen and producing a 1000px+ wall of rows. Compact date
formatting + multi-column layouts + a wider dashboard shell fix this without
touching any data, query, action, or business logic.

Output: eight modified files. PURE presentation/layout — NO change to server
actions, queries, data flow, vote/results computation, the three-token access
discipline, or any schema. NO migration. NO deploy (the orchestrator
re-screenshots admin + participant at desktop+mobile and deploys afterward).
</objective>
