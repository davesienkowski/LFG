## Phase Goal (MVP user story)

**As a** participant who already responded, **I want to** return via my personal edit link or the same device and change only my own availability while the poll is open, **so that** I can update my answer when plans change — and no one else can edit it for me.

This is the second vertical slice of Phase 2: after it ships, the full return-and-edit journey works end-to-end and Phase 2 is live in production.

<scope_note>
**Edge-probe resolution carried from 02-01 (VOTE-01/VOTE-05 idempotency — same-device re-submit):** A same-device re-submit MUST update the existing participant, never create a duplicate (02-SPEC Interview Log Round 2 + D-05). This plan owns that resolution: the participant page reads the same-device cookie and, when it resolves to a participant of this poll, preloads the prior response AND points the shared VoteForm at updateResponse (carrying the editToken) — so the re-submit upserts the same participant's rows. submitResponse (02-01) remains insert-only. The edit-link route uses the SAME updateResponse, sourcing the token from the URL instead of the cookie.
</scope_note>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| editor browser -> updateResponse | An untrusted caller presents an editToken (via URL or same-device cookie) — an unauthenticated BEARER credential — to mutate a stored response |
| edit route / participant RSC -> Postgres | Ownership is decided server-side by exact token lookup; the route renders read-only on closed polls but the server is the authority |
| deploy -> production Neon + Vercel | New tables must reach Neon before code serves traffic |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Elevation / Tampering (IDOR) | updateResponse ownership | mitigate | Re-derive the participant from the server-validated editToken via exact eq() lookup; cross-check participant.pollId === poll.id; never trust a client participantId (VOTE-06). 126-bit nanoid defeats token guessing — no constant-time compare needed (Pitfall 5) |
| T-02-07 | Tampering (TOCTOU) | poll.status guard | mitigate | Re-fetch poll.status at write time and reject when != 'open' on both updateResponse and the same-device re-submit path |
| T-02-08 | Information Disclosure | edit-route 404 surface | mitigate | Identical notFound() for a malformed token and a valid-but-unknown token — no token-format oracle (UI prohibition test) |
| T-02-09 | Tampering (race) | concurrent edits | mitigate | Single onConflictDoUpdate atomic statement → last-write-wins, never a mixed/partial set (concurrency backstop test) |
| T-02-10 | Tampering (SQLi) | sql`excluded.state` | mitigate | Fixed literal referencing the Postgres excluded pseudo-table (no interpolation); state is Zod-enum-validated; parameterized .values() is the actual defense |
| T-02-11 | Denial of Service (prod 500) | Neon schema drift | mitigate | BLOCKING db:migrate against Neon before the redeploy serves traffic (RESEARCH Pitfall 6) |
| T-02-SC | Tampering (supply chain) | npm installs | accept | Zero new packages this phase (RESEARCH Package Legitimacy Audit: N/A) |
