# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: Results Dashboard.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Continuity note:** Reuses the exact `base-nova / neutral / base-ui` preset and
> the Phase 1/2 spacing/typography/color tokens verbatim — this phase adds zero
> new shadcn components, zero new runtime dependencies (per RESEARCH.md), and
> zero new design tokens. The only genuinely new visual vocabulary is the
> best-day badge/column tint and the compact read-only cell chip, both built
> from the existing three-state (`STATE_META`) palette already locked in Phase 2.

**No new shadcn components required for Phase 3.** Reuse `button` and `card` only. The date/status filter uses **native `<select>`** elements (not a shadcn `Select`, which is not installed) — native selects are fully keyboard- and screen-reader-operable out of the box and match the project's existing precedent of preferring native form controls where they satisfy the requirement (Phase 1 used native `<input type="date">` over a Calendar for the same reason). `BestDayBadge` and the active-filter chip are hand-styled `<span>` pills — the exact pattern already used for Phase 1's "Keep private" badge, not an installed shadcn `Badge`.

| `button` | Results section | "Clear filter" button (reused, `variant="ghost"` or `variant="outline"`) |
| `card` | Not used this phase | The results table is NOT wrapped in a `Card` — a full-bleed `overflow-x-auto` table reads better at grid widths than a padded card interior; matches D-04's `overflow-x-auto` container literally |

| `<select>` (native) | Filter control | Date selector + status selector (D-06) |

## Spacing Scale

Reused verbatim from Phase 1/2 — no new tokens. All values are multiples of 4.

| Token | Value | Tailwind class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `gap-1` / `p-1` | Icon-to-label gap inside a cell chip, chip internal gap |
| sm | 8px | `gap-2` / `p-2` | Table cell vertical padding (`py-2`), filter control gap |
| md | 16px | `gap-4` / `p-4` | Section gap (heading → filter → table) |
| lg | 24px | `gap-6` / `p-6` | Section padding, matches existing page rhythm |
| xl | 32px | `gap-8` / `p-8` | Not used this phase |
| 2xl | 48px | `gap-12` / `p-12` | Page-level vertical padding (inherited from the existing `<main>` container — unchanged) |

**Exceptions (new for Phase 3):**

- Table cell horizontal padding: `px-3` (12px — a multiple of 4) — tighter than the 16px default because a dense grid (up to ~6 participants × several date columns) needs to stay legible without excessive horizontal scroll.
- Filter `<select>` elements and the "Clear filter" button: minimum height **44px** (`h-11`), same touch-target override Phase 1/2 already apply to every interactive form control on this project.
- Cell state chip (read-only, inside `<td>`): compact `px-2 py-0.5` — smaller than the interactive `AvailabilityGrid` button (`min-h-12`) because this chip is **not a touch target** (read-only, non-interactive `<span>`); the touch-target rule only applies to actionable controls.
- Sticky first column (`<th scope="row">`): `z-10` layering, no spacing change — a stacking-context exception, not a spacing exception.

## Typography

**Reused verbatim from Phase 1/2 — same 5 roles (4 carried forward + 1 formalized this phase), same 2 weights. No new sizes or weights introduced; the 5th size already ships in production.**

| Role | Size | Weight | Line Height | Tailwind | Usage |
|------|------|--------|-------------|---------|-------|
| Display | 30px | 600 (semibold) | 1.1 | `text-3xl font-semibold leading-tight` | Poll title (unchanged, above this phase's additions) |
| Heading | 24px | 600 (semibold) | 1.2 | `text-2xl font-semibold leading-snug` | "Results" section heading, empty-state heading |
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-relaxed` | Empty-state body copy, zero-match message body |
| Label | 14px | 600 (semibold) | 1.5 | `text-sm font-semibold leading-normal` | Filter field labels, participant name (row header), column date label |
| Caption / Badge | 12px | 400 or 600 (regular or semibold) | 1.5 | `text-xs font-normal` / `text-xs font-semibold` | Tally caption (`"{yes} yes · {ifneedbe} if-need-be"`), `BestDayBadge` pill text, active-filter chip text, cell state chip text |

**Caption / Badge role — carried forward from Phase 1, formalized this phase.** `text-xs` (12px) is not new UI vocabulary: it already ships in production on Phase 1's "Keep private" badge (`src/app/a/[adminUrlId]/page.tsx:85` — `text-xs font-semibold`). Phase 3 reuses this exact size for its own secondary/caption/badge text (tally caption, `BestDayBadge`, active-filter chip, cell state chip) rather than inventing a new size, so this contract formally declares it as the project's 5th typography role instead of leaving it an undocumented exception. The weight budget stays at 2 (regular 400 + semibold 600) — this role only ever borrows those two existing weights, never introduces a third. Scope is intentionally narrow: 12px is reserved exclusively for secondary/caption/badge text and is never used for body copy or headings, so the role addition doesn't erode the heading/body hierarchy.

Font stack: `Inter, system-ui, -apple-system, sans-serif` (unchanged).

## Component Spec: `ResultsGrid`

The core net-new component (SPEC DASH-01..05; CONTEXT D-04..07).

### Structure

```
<ResultsGrid>
  ┌─ Filter control (D-06) ──────────────────────────────────────┐
  │ Date [Choose a date… ▾]   Status [Available ▾]  [Clear filter]│
  │ (active-filter chip)              {count} of {total} shown    │
  └──────────────────────────────────────────────────────────────┘
  ┌─ overflow-x-auto (scroll-edge fade, see below) ─────────────────┐
  │ <table>                                                        │
  │  <caption class="sr-only">Availability results by candidate    │
  │   date</caption>                                                │
  │  <thead><tr>                                                   │
  │   <th scope="col" (sticky)>Participant</th>                    │
  │   <th scope="col">{date label}\n[Best]\n{yes} yes · {if} if}   │
  │   ...                                                           │
  │  </tr></thead>                                                 │
  │  <tbody>                                                        │
  │   <tr><th scope="row" (sticky)>{name}</th><td>chip</td>...</tr>│
  │   ...                                                           │
  │  </tbody>                                                       │
  │ </table>                                                        │
  └──────────────────────────────────────────────────────────────┘
</ResultsGrid>
```

- Table columns render in `getOptionsForPoll` chronological order (D-05) — no re-sort, mirrors Phase 2's `AvailabilityGrid` row order convention applied to columns here.
- Rows render in `getResultsForPoll` order — `participants.createdAt asc` (submission order), same "no re-sort, trust the query" convention.
- **Sticky first column:** `<th scope="row">` (and the corner `<th scope="col">Participant</th>`) use `sticky left-0 z-10 bg-background border-r` so the participant name stays visible while horizontally scrolling on mobile (`overflow-x-auto` container, D-04). This is the Claude's-Discretion sticky-column implementation the RESEARCH.md/CONTEXT.md left open — resolved here as the default.
- Row striping: `tbody tr:nth-child(even)` gets `bg-muted/40` for readability at up to ~10 rows — a minimal, low-risk addition consistent with the "calm, minimal" existing pages.
- **Scroll-edge affordance (prohibition-probe finding — see Prohibitions table below):** the `overflow-x-auto` wrapper carries a right-edge fade (`mask-image: linear-gradient(to right, black calc(100% - 24px), transparent)` or an equivalent `after:` gradient overlay) whenever the table's scrollable width exceeds its visible width, so a viewer on a narrow screen sees a visual cue that additional date columns — potentially including the `BestDayBadge` column — exist off-screen. Without this, the single most decision-relevant element on the page (which day is "Best") could scroll out of view with zero indication there was more to see. The fade is a pure CSS affordance (no new dependency); a plain `overflow-x-auto` with no cue is not acceptable per this finding.

### Column header anatomy (`<th scope="col">`)

```tsx
<th scope="col" className={cn("px-3 py-2 align-bottom text-center", isBest && "bg-emerald-50")}>
  <div className="flex flex-col items-center gap-1">
    <span className="text-sm font-semibold whitespace-nowrap">{dateLabel}</span>
    {isBest && <BestDayBadge />}
    <span className="text-xs font-normal text-muted-foreground whitespace-nowrap">
      {yes} yes · {ifneedbe} if-need-be
    </span>
  </div>
</th>
```

- `dateLabel` via `formatDateWithTime` (timezone-safe, D-11/P3) — identical helper/format Phase 1/2 already use.
- Tally caption is **always rendered**, including `0 yes · 0 if-need-be` for a zero-vote date (SPEC AC-4) — never hidden or omitted at zero.
- `BestDayBadge` renders only when `isBest === true` for that column (computed by `computeResults`, D-02); **all** co-leading columns render it (never collapsed to one).

### Row header anatomy (`<th scope="row">`)

```tsx
<th scope="row" className="sticky left-0 z-10 border-r bg-background px-3 py-2 text-left text-sm font-semibold whitespace-nowrap">
  {participant.name}
</th>
```

Two participants sharing a display name render as two distinct rows (keyed by participant `id`, SPEC AC-1) — the visible text can be identical; uniqueness is structural (row position + `key`), never a UI concern to disambiguate visually.

### Cell anatomy (`<td>`, read-only chip)

```tsx
<td className={cn("px-3 py-2 text-center", isBest && "bg-emerald-50/50")}>
  <span className={cn(
    "inline-flex items-center gap-1 rounded-full border px-2 py-0.5 text-xs font-semibold",
    STATE_META[state].className,
  )}>
    <Icon aria-hidden className="size-3.5" />
    {STATE_META[state].label}
  </span>
</td>
```

- `state = normalizeVoteState(participant.votes[optionId])` — routes through the shared gap-fill/fallback helper (D-03); a missing vote or unrecognized literal renders "Not available" (`X` icon, neutral chip), never throws, never blank.
- **Color is never the only signal** — every chip renders `Icon` (`aria-hidden`) **and** the visible text label together, identical invariant to Phase 2's `AvailabilityGrid`.
- Not a touch target (read-only `<span>`, not a `<button>`) — no `min-h-*` requirement; sized for table density instead (see Spacing exceptions).

### Empty state (zero participants — SPEC AC-1/AC-7)

Replaces the entire table (not rendered alongside a broken/empty grid), styled like Phase 1's placeholder banner:

```tsx
<div className="rounded-lg border bg-muted p-6 text-center">
  <p className="text-2xl font-semibold leading-snug">No responses yet</p>
  <p className="mt-2 text-base text-muted-foreground">
    No one has responded yet. Share the participant link above to start collecting availability.
  </p>
</div>
```

The filter control is **not rendered** in this state (nothing to filter).

## Component Spec: Filter Control (DASH-05, D-06)

A `"use client"` island holding local `useState<{ dateId: string | null; status: VoteState }>`. **No network round-trip** on any interaction (D-06) — filters the already-rendered participant array in memory.

### Structure

```tsx
<div className="flex flex-col gap-3">
  <div className="flex flex-wrap items-end gap-3">
    <div className="flex flex-col gap-1">
      <label htmlFor="results-filter-date" className="text-sm font-semibold">Date</label>
      <select
        id="results-filter-date"
        className="h-11 rounded-lg border border-input bg-background px-3 text-base"
        value={dateId ?? ""}
        onChange={...}
      >
        <option value="">Choose a date…</option>
        {options.map((o) => <option key={o.id} value={o.id}>{formatDateWithTime(o.date, o.startTime)}</option>)}
      </select>
    </div>
    <div className="flex flex-col gap-1">
      <label htmlFor="results-filter-status" className="text-sm font-semibold">Status</label>
      <select
        id="results-filter-status"
        className="h-11 rounded-lg border border-input bg-background px-3 text-base"
        value={status}
        onChange={...}
      >
        <option value="yes">Available</option>
        <option value="ifneedbe">If-need-be</option>
        <option value="no">Not available</option>
      </select>
    </div>
    <Button type="button" variant="ghost" className="h-11" disabled={!dateId} onClick={clear}>
      <X aria-hidden />
      Clear filter
    </Button>
  </div>

  {dateId && (
    <div className="flex flex-wrap items-center gap-2">
      <span className="inline-flex items-center rounded-full border bg-muted px-2 py-0.5 text-xs font-semibold">
        {selectedDateLabel} · {STATE_META[status].label}
      </span>
      <span className="text-sm text-muted-foreground">{matchCount} of {totalCount} participants</span>
    </div>
  )}

  {/* sr-only live region, mirrors AvailabilityGrid's announcement pattern */}
  <div aria-live="polite" className="sr-only">{filterAnnouncement}</div>
</div>
```

### Behavior

- **Status defaults to "Available"** (`yes`) — the organizer's most common question is "who's available on date X" (RESEARCH.md/CONTEXT rationale). **Date defaults to unselected** (placeholder "Choose a date…").
- The filter engages the moment a date is selected (using whatever status is currently selected, default "Available") — a single interaction (pick a date) is enough to see a filtered result; changing the status re-filters instantly. No separate "Apply" button (matches D-06: filter reacts to selection, not a submit action).
- **"Clear filter"** resets `dateId` to `null` (status may stay at its last value or reset to "Available" — either is acceptable; resetting to "Available" keeps behavior predictable). Disabled/inert until a date is selected (nothing to clear beforehand).
- **Filtering hides non-matching rows** (D-06: "Filter-hide chosen over sort-to-top") — the `<tbody>` renders only participant rows whose `state` for the selected date equals the selected status; the table header/columns are unaffected (all date columns and tallies keep displaying, only participant *rows* filter).
- **Zero matches:** the table's `<tbody>` is replaced with a single full-width message row (not a blank table, not a missing table) — see below.
- The active-filter chip and result count (`"{count} of {total} participants"`) render only while a date is selected; disappear entirely when cleared.

### Zero-match state (SPEC AC-6)

```tsx
<tbody>
  <tr>
    <td colSpan={options.length + 1} className="px-3 py-8 text-center">
      <p className="text-2xl font-semibold leading-snug">No participants match</p>
      <p className="mt-2 text-base text-muted-foreground">
        Try a different date or status, or clear the filter.
      </p>
    </td>
  </tr>
</tbody>
```

**Layout:** Same centered container as the rest of the admin page (`max-w-2xl mx-auto` is the existing constraint — **note:** the results table is the first element on this page that may need to exceed 640px on wide viewports since a grid with several date columns can be wider than the text content above it; the table's own `overflow-x-auto` wrapper handles this at all viewport widths rather than widening the whole page container, keeping the rest of the page's reading-width consistent).

## Interaction States

### Filter — Selection Change

- Selecting a date immediately filters the visible rows (client `useState`, synchronous re-render) — no loading spinner, no debounce needed (all-in-memory, D&D-group scale).
- Changing status while a date is already selected immediately re-filters.
- The `sr-only aria-live="polite"` region announces: `"{count} of {total} participants shown for {date label}, {status label}"` on every filter change — mirrors the `AvailabilityGrid` cell-change announcement pattern from Phase 2.

### Filter — Clear

- Clicking "Clear filter" (or re-selecting the date placeholder "Choose a date…") removes the filter; all participant rows reappear; the active-filter chip and count disappear; the live region announces `"Showing all {total} participants"`.

### Best-Day Highlight — No Winner

- When no date has ≥1 `yes` vote (all-zero-yes, SPEC AC-5), **no** column renders `BestDayBadge` and **no** column gets the emerald tint — the grid renders as a plain, unhighlighted table. This is a valid, expected state (e.g., immediately after the poll opens with zero votes), not an error.

### Table — Zero Participants vs. Zero Filter Matches

These are two distinct, non-interchangeable states — do not conflate their copy:

| State | Trigger | Copy |
|-------|---------|------|
| Zero participants | `getResultsForPoll` returns an empty array | "No responses yet" (banner replaces the entire table + filter control) |
| Zero filter matches | Participants exist, but none match the selected date+status | "No participants match" (table renders normally; only the `<tbody>` shows this row; filter control stays visible so the organizer can adjust it) |

## Accessibility Requirements

- Table uses proper semantic structure: `<th scope="col">` for every date column header (including the corner "Participant" header) and `<th scope="row">` for every participant name — screen readers announce row/column context automatically when navigating cells (D-04).
- `<caption class="sr-only">` on the table gives assistive tech a one-line summary of the table's purpose before diving into cells.
- **Every cell communicates state via icon + text label, never color alone** — locked invariant, identical to Phase 2's `AvailabilityGrid` (STATE_META reuse, D-05).
- **The best-day highlight is never color-only** — the "Best" text label always accompanies the emerald tint (D-07); a colorblind or screen-reader user gets the same information as a sighted user scanning for color.
- Filter `<select>` elements have associated `<label htmlFor>` pairing (native semantics — fully keyboard operable via arrow keys/typeahead without any extra ARIA).
- The "Clear filter" button is disabled (not merely styled dim) until a date is selected — `disabled` attribute prevents a no-op activation from confusing keyboard/AT users.
- A `sr-only aria-live="polite"` region announces filter-result changes (count + active date/status) — compensates for the filtered `<tbody>` update not otherwise producing a screen-reader announcement (same rationale as Phase 2's cell-change live region).
- Minimum touch target: 44px (`h-11`) for both `<select>` elements and the "Clear filter" button — same override pattern already applied throughout Phase 1/2 (native selects render below 44px by default in most browsers).
- Sticky first column (`z-10`) does not reduce the visible content of scrolled-under cells — it only pins the name column; horizontal scroll (`overflow-x-auto`) remains available via touch/trackpad/keyboard (native scrollable-region focus + arrow keys once focused).
- Focus order: Date select → Status select → Clear filter button → table (browsers navigate table cells via standard DOM order, top-to-bottom, left-to-right — no custom tab-stops on non-interactive cells, since chips are read-only `<span>`s, not focusable elements).
- Cell chip colors (`emerald-50/700`, `amber-50/700`, `muted/muted-foreground`) are the exact palette Phase 2 already verified clears WCAG AA text contrast (4.5:1) — no new contrast verification needed, reused verbatim.

## Prohibitions (UI — prohibition-probe)

**Probe policy:** SPEC.md already resolved the requirements-level prohibitions (no participant email in output, no results capability on the participant surface, no token leakage) — these are server/query-level concerns owned by SPEC.md and are NOT re-probed here. This section runs the prohibition-probe over the genuinely NEW UI behaviors this contract adds — the read-only cell chip, the best-day highlight, and the client-side filter — keeping only bespoke value/transparency/accessibility items; routine engineering candidates already covered by Phase 1/2 precedent (e.g., "Clear filter needs no confirmation dialog" — already a reversible, non-persistent, low-stakes action) were dropped.

| Prohibition (must-NOT statement) | Surface | Status | Verification |
|----------------------------------|---------|--------|--------------|
| MUST NOT communicate any cell's vote state, or the best-day highlight, using color as the only signal | `ResultsGrid` cells + `BestDayBadge` | resolved | test — assert every cell renders both a lucide icon AND a visible text label; assert every `isBest` column header renders the literal "Best" text alongside the emerald tint class |
| MUST NOT trigger a network request, Server Action, or `router.refresh()` when the filter selection changes | Filter control (`ResultsGrid` island) | resolved | test — assert selecting a date/status option does not invoke `fetch`/a Server Action; the filtered row set changes synchronously from already-rendered props |
| MUST NOT render a blank or structurally broken table when participants is empty, or when a filter matches zero rows | `ResultsGrid` | resolved | test — assert `participants.length === 0` renders the "No responses yet" banner (table absent) and a zero-match filter renders the "No participants match" `<tbody>` row (table present, headers intact) |
| MUST NOT render the tally caption as blank/omitted for a zero-vote date | Column header | resolved | test — assert a date with 0 yes / 0 if-need-be still renders the literal text "0 yes · 0 if-need-be", not an empty string |
| MUST NOT collapse tied co-leading best-day columns to a single highlighted column | Column header / `BestDayBadge` | resolved | test — assert two dates tied on `(yes, ifneedbe)` both render `BestDayBadge` and both get the emerald column tint |
| MUST NOT let the results table's horizontal scroll hide the best-day badge/column with no visual indication that additional columns exist off-screen — this would silently defeat the core purpose of DASH-04's best-day highlight on any viewport narrower than the full column set | `ResultsGrid` `overflow-x-auto` container | resolved | judgment — a scroll-edge fade affordance (see Component Spec above) is a visual-QA item, not mechanically unit-testable; verified by manual/visual review that the fade renders whenever scrollable width exceeds visible width |
| MUST NOT display `BestDayBadge` without its supporting yes/if-need-be tally visible in the same column header — a badge alone could imply broad consensus that a small sample size doesn't actually support | Column header (`<th scope="col">`) | resolved | test — assert every `isBest` column's header renders both the `BestDayBadge` AND the tally caption text in the same `<th>`, never the badge alone |

**Dismissed (considered, not minted):** exceeding 640px page width for the table on desktop was raised in Stage 1 but is NOT a UI omission — it's an intentional, documented layout decision (see Surface section: the table's own `overflow-x-auto` wrapper handles width, the outer page container stays unchanged) rather than a prohibition needing a negative test. A "Best" badge being misread as a finalized/booked decision was also raised in Stage 1 and dismissed — the copy is strictly "Best" (a ranking label), never "Confirmed"/"Booked"/"Selected", and Phase 4 owns the actual finalize/"Book it" UI; no ambiguous copy exists in this contract to correct.

These become `must_haves` for the planner/verifier (negative acceptance criteria), in addition to SPEC.md's own Prohibitions table.
