- **D-10:** Absolute share URLs built from `NEXT_PUBLIC_BASE_URL` env, falling back to request headers when unset. Admin page renders both URLs with copy-to-clipboard, labelled "Participant link" and "Admin link (keep private)".
- **D-11:** Date-only values stored as Postgres `DATE`, handled in app code as `'YYYY-MM-DD'` strings. NEVER `new Date("YYYY-MM-DD")`. A test renders a date under TZ `Pacific/Kiritimati` (UTC+14) and `Etc/GMT+12` (UTC−12) and asserts identical calendar day.

## Summary

Phase 1 is a greenfield walking-skeleton: scaffold Next.js 16 → wire Drizzle ORM → write the `polls` + `options` schema → implement the `createPoll` server action → render the creation form and admin page with two share links. Everything downstream (voting, dashboard, email) builds on the schema and token model established here — these decisions are the highest cost-to-change in the entire project.

The research confirms all decisions in D-01 through D-11 are sound and directly implementable with the locked stack. Key implementation-ready specifics: the `create-next-app` invocation installs Next.js 16 (current `latest`); Drizzle's `date({ mode: "string" })` returns YYYY-MM-DD strings natively (no transformation needed); `unique().on(...).nullsNotDistinct()` is the correct Drizzle API for the nullable `start_time` dedup constraint on Postgres 15+; and the dual-driver DB client switches via `process.env.NODE_ENV`.

The primary implementation risks are the nullable `start_time` unique constraint (requires `NULLS NOT DISTINCT`, Postgres 15+ feature, verified available on both postgres:17 Docker and Neon), and the Vercel function timeout for Neon cold-start (mitigated by using the pooled connection string with ≥10s timeout).

**Primary recommendation:** Follow the vertical-slice build order — scaffold and Docker Compose first, schema + migrations second, createPoll action third, creation form and admin page last — so each step is testable before the next begins.

### Pattern 1: Dual-Driver Drizzle Client (D-03)

**What:** Single `db` export selects `node-postgres` locally and `neon-http` on Vercel based on `NODE_ENV`.

**Why:** Next.js sets `NODE_ENV=production` on Vercel automatically. No manual env switching needed.

**Example:**

```typescript
// Source: STACK.md + Drizzle official docs (orm.drizzle.team/docs/get-started/neon-new)
// src/lib/db/index.ts
import { drizzle } from "drizzle-orm/node-postgres";
import { drizzle as neonDrizzle } from "drizzle-orm/neon-http";
import { neon } from "@neondatabase/serverless";
import * as schema from "./schema";

export const db =
  process.env.NODE_ENV === "production"
    ? neonDrizzle(neon(process.env.DATABASE_URL!), { schema })
    : drizzle(process.env.DATABASE_URL!, { schema });
```

**Note on types:** Both drivers return a compatible Drizzle `db` object; TypeScript infers `DB` as a union type. For ergonomics, declare the type explicitly:

```typescript
import type { NodePgDatabase } from "drizzle-orm/node-postgres";
import type { NeonHttpDatabase } from "drizzle-orm/neon-http";
import type * as schema from "./schema";

type DB = NodePgDatabase<typeof schema> | NeonHttpDatabase<typeof schema>;
export let db: DB;
// ... assign as above
```

### Pattern 2: Drizzle Schema (D-05, D-06)

**Critical detail:** Use `date({ mode: "string" })` — this makes Drizzle return `YYYY-MM-DD` strings directly from Postgres, bypassing JavaScript Date parsing entirely. This is the correct implementation of D-11.

**Critical detail:** Use `.nullsNotDistinct()` on the `(poll_id, date, start_time)` unique constraint. Without it, two date-only options (both with `start_time = NULL`) would NOT be caught as duplicates by Postgres (SQL standard: `NULL != NULL` in unique constraints). `NULLS NOT DISTINCT` is available in Postgres 15+ — both `postgres:17` (local) and Neon (Postgres 17) qualify. [VERIFIED: Drizzle docs orm.drizzle.team/docs/indexes-constraints]

```typescript
// Source: Drizzle official docs (orm.drizzle.team/docs/column-types/pg,
//         orm.drizzle.team/docs/indexes-constraints)
// src/lib/db/schema.ts
import {
  pgTable, uuid, text, integer, time, date, timestamp, unique, index
} from "drizzle-orm/pg-core";

export const polls = pgTable("polls", {
  id: uuid("id").primaryKey().defaultRandom(),
  participantUrlId: text("participant_url_id").notNull().unique(),
  adminUrlId: text("admin_url_id").notNull().unique(),
  title: text("title").notNull(),
  description: text("description"),
  location: text("location"),
  status: text("status").notNull().default("open"),
  createdAt: timestamp("created_at", { withTimezone: true })
    .notNull()
    .defaultNow(),
});

export const options = pgTable(
  "options",
  {
    id: uuid("id").primaryKey().defaultRandom(),
    pollId: uuid("poll_id")
      .notNull()
      .references(() => polls.id, { onDelete: "cascade" }),
    date: date("date", { mode: "string" }).notNull(),  // returns 'YYYY-MM-DD' string
    startTime: time("start_time"),                     // nullable — date-only is valid
    position: integer("position").notNull().default(0),
  },
  (t) => [
    // NULLS NOT DISTINCT: treats two NULLs as equal — catches duplicate date-only entries
    unique("options_dedup").on(t.pollId, t.date, t.startTime).nullsNotDistinct(),
    index("options_poll_id_idx").on(t.pollId),
  ]
);

// Type exports for use in server actions
export type Poll = typeof polls.$inferSelect;
export type NewPoll = typeof polls.$inferInsert;
export type Option = typeof options.$inferSelect;
export type NewOption = typeof options.$inferInsert;
```

### Pattern 3: createPoll Server Action (D-07, D-08)

**What:** `'use server'` function validates with Zod, generates two independent tokens, inserts poll + options, redirects to admin page.

**Key points:**
- `redirect()` throws an exception — code after it does not run; no `return` needed after redirect.
- Both tokens generated by separate `nanoid(21)` calls — never derive one from the other.
- Collision retry: with 2^126 entropy space, collisions are astronomically improbable; a try/catch on the DB unique constraint violation is sufficient.
- `useActionState` (React 19, imported from `react`) wraps the action for form error display.

```typescript
// Source: Next.js official docs (nextjs.org/docs/app/getting-started/mutating-data)
// src/lib/actions/create-poll.ts
"use server";

import { z } from "zod";
import { redirect } from "next/navigation";
import { db } from "@/lib/db";
import { polls, options } from "@/lib/db/schema";
import { nanoid } from "nanoid";

const DateOptionSchema = z.object({
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/, "Invalid date format"),
  startTime: z.string().regex(/^\d{2}:\d{2}$/).optional().nullable(),
});

const CreatePollSchema = z.object({
  title: z.string().min(1, "Title is required").trim(),
  description: z.string().max(2000).optional(),
  location: z.string().max(200).optional(),
  // dates arrive as JSON string from hidden input (see form component)
  dates: z.string().transform((val) => {
    const parsed = JSON.parse(val);
    return DateOptionSchema.array().min(1, "At least one date required").parse(parsed);
  }),
});

export type CreatePollState = {
  errors?: Record<string, string[]>;
} | null;

export async function createPoll(
  _prevState: CreatePollState,
  formData: FormData
): Promise<CreatePollState> {
  const raw = {
    title: formData.get("title"),
    description: formData.get("description") || undefined,
    location: formData.get("location") || undefined,
    dates: formData.get("dates"),
  };

  const parsed = CreatePollSchema.safeParse(raw);
  if (!parsed.success) {
    return { errors: parsed.error.flatten().fieldErrors };
  }

  const { title, description, location, dates } = parsed.data;

  // Deduplicate dates at application layer (belt-and-suspenders before DB constraint)
  const uniqueDates = dates.filter(
    (d, i, arr) =>
      arr.findIndex(
        (o) => o.date === d.date && (o.startTime ?? null) === (d.startTime ?? null)
      ) === i
  );

  // Two independent crypto-random tokens — never derive one from the other (D-07 / P1)
  const participantUrlId = nanoid(21);
  const adminUrlId = nanoid(21);

  const [poll] = await db
    .insert(polls)
    .values({ title, description, location, participantUrlId, adminUrlId })
    .returning({ id: polls.id });

  await db.insert(options).values(
    uniqueDates.map((d, i) => ({
      pollId: poll.id,
      date: d.date,
      startTime: d.startTime ?? null,
      position: i,
    }))
  );

  // redirect() throws — no return after this line
  redirect(`/a/${adminUrlId}`);
}
```

### Pattern 4: Timezone-Safe Date Rendering (D-11 / PLAT-04)

**The footgun:** `new Date("2025-07-12")` parses as UTC midnight. In any UTC-offset timezone (e.g., UTC-4), this displays as July 11. This bug affects anyone whose browser timezone is behind UTC — which includes all of North America.

**The fix:** Never construct a `Date` object from a date-only string. Treat the date as a string throughout.

```typescript
// WRONG — shifts the calendar day in negative-offset timezones:
const d = new Date("2025-07-12");
d.toLocaleDateString(); // → "7/11/2025" in UTC-4

// CORRECT — string manipulation, timezone-immune:
function formatDateString(yyyymmdd: string): string {
  const [year, month, day] = yyyymmdd.split("-");
  // Return a locale-friendly string without going through Date
  return `${parseInt(month)}/${parseInt(day)}/${year}`;
  // Or use a library: parseISO from date-fns parses date-only as LOCAL date, not UTC
}

// If using date-fns (optional dep, Claude's discretion):
import { parseISO, format } from "date-fns";
// parseISO("2025-07-12") → treated as local midnight, not UTC midnight
const formatted = format(parseISO("2025-07-12"), "MMM d, yyyy");
```

**TZ test pattern (implements D-11 acceptance criterion):**

```typescript
// Test: same YYYY-MM-DD renders same calendar day in UTC+14 and UTC−12
// Run with: TZ=Pacific/Kiritimati node -e "..." and TZ=Etc/GMT+12 node -e "..."

function formatDateNoDateConstructor(yyyymmdd: string): string {
  const [y, m, d] = yyyymmdd.split("-");
  return `${y}-${m}-${d}`; // passthrough — same string regardless of TZ
}

// The test verifies this function returns identical output regardless of TZ env var.
// Compare: formatDateNoDateConstructor("2025-07-12") under both TZ values → must match.
```

**Drizzle returns date as string natively:** Because the schema uses `date("date", { mode: "string" })`, the DB driver never constructs a JavaScript `Date` object for the date column. The value arrives in server-side code as the string `"2025-07-12"` directly from Postgres. Pass this string to the client and format with string operations — no Date construction required at any tier.

### Pattern 7: Admin Page — Both Share Links (D-08, D-09, D-10)

```typescript
// Source: Next.js App Router RSC pattern
// src/app/a/[adminUrlId]/page.tsx
import { notFound } from "next/navigation";
import { db } from "@/lib/db";
import { polls, options } from "@/lib/db/schema";
import { eq } from "drizzle-orm";
import { headers } from "next/headers";

export default async function AdminPage({
  params,
}: {
  params: Promise<{ adminUrlId: string }>;
}) {
  const { adminUrlId } = await params;

  const poll = await db
    .select()
    .from(polls)
    .where(eq(polls.adminUrlId, adminUrlId))
    .limit(1);

  if (!poll.length) notFound();

  // Build absolute URLs (D-10): prefer NEXT_PUBLIC_BASE_URL, fallback to host header
  const headersList = await headers();
  const host =
    process.env.NEXT_PUBLIC_BASE_URL ??
    `https://${headersList.get("host")}`;

  const participantUrl = `${host}/p/${poll[0].participantUrlId}`;
  const adminUrl = `${host}/a/${adminUrlId}`;

  return (
    <div>
      <h1>{poll[0].title}</h1>
      {/* Participant link — safe to share */}
      <p>Participant link: {participantUrl}</p>
      {/* Admin link — marked private (D-10 + SPEC requirement) */}
      <p>Admin link (keep private): {adminUrl}</p>
      {/* CopyLinkButton is a 'use client' component */}
    </div>
  );
}
```

### Anti-Patterns to Avoid

- **Deriving admin token from participant token:** Never `adminUrlId = hash(participantUrlId)` or any transform. Use two independent `nanoid(21)` calls. (Prohibition P1)
- **Including adminUrlId in participant-facing queries or responses:** The participant route MUST NOT select or transmit `admin_url_id`. (Prohibition P2)
- **`new Date("YYYY-MM-DD")` anywhere:** Shifts calendar day in negative-offset timezones. Always split the string manually or use `date-fns`'s `parseISO`. (Prohibition P3, D-11)
- **Using `node-postgres` on Vercel:** `pg` uses TCP; Vercel serverless functions require HTTP/WebSocket. Use `@neondatabase/serverless` (neon-http) in production.
- **Using direct Neon connection string (non-pooled) for app queries:** The non-pooled string (`.neon.tech`, no `-pooler`) causes new TCP connection per function invocation. Use the pooled string (`-pooler.neon.tech`) for all app queries. Reserve direct connection for `drizzle-kit migrate` only.
- **Skipping `NULLS NOT DISTINCT` on the unique constraint:** Without it, two date-only options with the same date are NOT caught as duplicates (SQL NULL != NULL rule). Use `unique().on(...).nullsNotDistinct()`.
- **`sequential`/`integer` PKs in URLs:** Auto-increment integers are enumerable. Always use UUID PKs internally; use `participant_url_id` / `admin_url_id` (nanoid) in URLs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Crypto-random token generation | Custom UUID-like string from `Math.random()` | `nanoid(21)` | `Math.random()` is not cryptographically secure; nanoid uses `crypto.getRandomValues()` |
| Schema validation in server actions | Manual `if (!title)` chains | `zod` with `.safeParse()` | Error messages, type narrowing, array validation, nested shapes — all free |
| Type-safe env var access | `process.env.DATABASE_URL as string` everywhere | `@t3-oss/env-nextjs` | Fails at build time with clear message rather than at runtime with undefined |
| Drizzle migration SQL | Hand-writing `CREATE TABLE` SQL | `drizzle-kit generate` + `drizzle-kit migrate` | Auto-diffs schema; handles column renames, constraint changes, index additions |
| Local Postgres for development | SQLite or in-memory mock | Docker Compose + `postgres:17` | Eliminates SQLite/Postgres dialect divergence; same engine as Neon in production |
| DB connection management | Manual `pg.Pool` singleton | Drizzle `db` client with environment-switched driver | Drizzle handles pooling for node-postgres; neon-http is stateless (correct for serverless) |

**Key insight:** In a serverless environment every byte of bundle and every millisecond of cold-start matters. Drizzle's 57 KB vs Prisma's 1.6 MB is a 28x difference. Do not add heavy dependencies for problems the locked stack already solves.

## Common Pitfalls

### Pitfall 1: NULL != NULL in Postgres Unique Constraints (date dedup)

**What goes wrong:** The composite unique constraint `(poll_id, date, start_time)` on the `options` table uses `start_time` which is nullable. In standard SQL (and Postgres default behavior), `NULL` is not equal to `NULL` in unique constraint evaluation. Two rows with `(same_poll_id, '2025-07-12', NULL)` would NOT be caught as duplicates by a default `UNIQUE` constraint — both inserts succeed, violating POLL-03.

**Why it happens:** Developers write `unique().on(t.pollId, t.date, t.startTime)` and assume it catches all duplicates. It doesn't when `start_time` is NULL.

**How to avoid:** Use `.nullsNotDistinct()` in Drizzle: `unique().on(t.pollId, t.date, t.startTime).nullsNotDistinct()`. This generates `CONSTRAINT ... UNIQUE (poll_id, date, start_time) NULLS NOT DISTINCT` — available in Postgres 15+. Also add application-level deduplication in `createPoll` before the DB insert as belt-and-suspenders. [VERIFIED: Drizzle docs orm.drizzle.team/docs/indexes-constraints]

**Warning signs:** Two date-only options with the same date appear in the `options` table after a single poll creation.

---

### Pitfall 2: Neon Cold-Start Exceeding Vercel Function Timeout

**What goes wrong:** Neon free tier suspends compute after 5 minutes of inactivity. Resume takes 1–3 seconds. If the app DB connection timeout is set to the default (usually 5s), the cold-start resume + query may exceed the timeout, returning a 504 to the user.

**Why it happens:** D&D group usage is sporadic — many real requests will be cold-start requests. Default driver timeouts are designed for always-on servers.

**How to avoid:**
- Always use the Neon **pooled** connection string (`-pooler.neon.tech`) for app queries. The pooler (PgBouncer) stays warm; only the Postgres compute sleeps.
- Set connection timeout ≥ 10 seconds in the neon-http client (or accept the serverless driver's default which handles retries).
- The Vercel Hobby function timeout is ≈60 seconds for Serverless Functions (extended in 2024 from the old 10s). [ASSUMED — verify against current Vercel docs; timeout values have changed historically]
- Test the cold-start path explicitly: let Neon idle for 10+ minutes, then hit the poll creation endpoint and verify it succeeds.

**Warning signs:** Intermittent 504 errors appearing only after periods of inactivity; never reproducible under load.

---

### Pitfall 3: `params` in Next.js 15/16 App Router is a Promise

**What goes wrong:** Next.js 15 changed `params` in page components from a synchronous object to a `Promise`. Code written for Next.js 14 (`params.adminUrlId`) fails with a type error in Next.js 15/16.

**Why it happens:** Migration guides for Next.js 14 → 15 show this change; generated code from older examples still uses the synchronous form.

**How to avoid:** Always `await params` in page components:

```typescript
export default async function AdminPage({
  params,
}: {
  params: Promise<{ adminUrlId: string }>;
}) {
  const { adminUrlId } = await params; // required in Next.js 15/16
  // ...
}
```

[CITED: nextjs.org — the official docs for Next.js 16 show `Promise<{ ... }>` for `params`]

**Warning signs:** TypeScript error `Property 'adminUrlId' does not exist on type 'Promise<...>'`.

---

### Pitfall 4: drizzle-orm and drizzle-kit Version Mismatch

**What goes wrong:** `drizzle-orm` and `drizzle-kit` must be kept at compatible versions. Mismatches cause silent migration failures or incorrect SQL generation.

**Why it happens:** Running `npm install drizzle-orm drizzle-kit` without pinning versions, then updating one independently.

**How to avoid:** Always update both together: `npm install drizzle-orm@x.y.z drizzle-kit@x.y.z`. Current compatible pair: `drizzle-orm@0.45.2` + `drizzle-kit@0.31.10`. [VERIFIED: STACK.md + npm registry version check]

**Warning signs:** `drizzle-kit generate` produces empty or incorrect SQL; `drizzle-kit push` errors on column types.

---

### Pitfall 5: Forgetting to Prefix with `NEXT_PUBLIC_` for Client-Accessible Env Vars

**What goes wrong:** `BASE_URL` without the `NEXT_PUBLIC_` prefix is available only in server-side code. The CopyLinkButton (a Client Component that displays the full URL) cannot read it on the client.

**How to avoid:** Use `NEXT_PUBLIC_BASE_URL` for the base URL env var. Server Actions and RSC pages can also read it. This is the pattern in D-10.

## Open Questions

1. **shadcn/ui date input for multi-date selection**
   - What we know: shadcn/ui ships a Calendar/DatePicker component (code-copied); it uses `date-fns` internally.
   - What's unclear: Whether the multi-date selection pattern needs a custom wrapper or if shadcn's DatePicker has multi-select built in.
   - Recommendation: Claude's Discretion applies. Use shadcn `Calendar` in `mode="multiple"` (it supports multi-select) or a simple list of text `<input type="date">` elements with add/remove buttons. The simpler `<input type="date">` approach is easier to implement for MVP and avoids a date library dependency.

2. **`input type="date"` timezone behavior in browsers**
   - What we know: `<input type="date">` returns `value` as a `YYYY-MM-DD` string — never as a Date object — so it is inherently timezone-safe when read as a string.
   - What's unclear: Whether different browser implementations might return the value in a non-standard format.
   - Recommendation: Use `input.value` (string) directly; validate format with Zod regex `^\d{4}-\d{2}-\d{2}$`. Do not call `new Date(input.value)`.

## Security Domain

Security enforcement is enabled (`security_enforcement: true`, `security_asvs_level: 1`).

### Applicable ASVS Categories (Level 1)

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No | No user accounts; link-based token access |
| V3 Session Management | No | No sessions; token validated per-request from DB |
| V4 Access Control | Yes | Admin vs participant route separation enforced server-side via DB lookup; participant route query NEVER selects `admin_url_id` |
| V5 Input Validation | Yes | Zod in `createPoll` server action; all fields validated server-side before any DB write |
| V6 Cryptography | Yes | `nanoid` uses `crypto.getRandomValues()` (CSPRNG); tokens are never derived or predictable |

### Known Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SQL injection via form fields | Tampering | Drizzle ORM uses parameterized queries at all times — no raw SQL in Phase 1 |
| Admin token exposure via participant route | Information Disclosure | Column projection in participant query excludes `admin_url_id` (D-09 / P2) |
| Enumeration of polls via sequential IDs | Information Disclosure | UUID PKs internally; `participant_url_id` / `admin_url_id` are nanoid(21) — 2^126 space |
| Admin token derived from participant token | Elevation of Privilege | Two independent `nanoid(21)` calls — no transform/derivation (D-07 / P1) |
| XSS via unescaped poll title/description | Tampering | React escapes all JSX string interpolation by default; no `dangerouslySetInnerHTML` needed |
| Server Action invocable by direct POST | Elevation of Privilege | Server Actions are POST-only and include a CSRF-equivalent challenge header automatically in Next.js 15+; no additional CSRF library needed for same-origin calls [CITED: nextjs.org/docs/app/getting-started/mutating-data] |

### Phase 1 Security Notes

- No authentication layer in this phase (by design — link-based access model).
- The admin token IS the authentication credential. Losing it means losing admin access to the poll. This is acceptable for a single-group D&D tool.
- Tokens in URLs appear in server access logs and browser history. Acceptable for a private group tool; note to user if deploying publicly.
- Participant emails (Phase 4) stored in Postgres — note for future GDPR consideration. Not in scope for Phase 1.
