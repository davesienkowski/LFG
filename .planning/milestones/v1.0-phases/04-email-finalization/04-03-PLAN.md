## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Executor → Neon (prod DB) | The pulled Neon DATABASE_URL is a production credential; used once to apply the additive migration |
| Executor → Vercel (deploy + env) | Production deploy + env var read; the App Password is a Google credential held in Vercel |
| Prod runtime → Gmail SMTP | The App Password authenticates outbound mail as the owner's gmail (self-aligning SPF/DKIM/DMARC, D-01/D-03) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-12 | Information Disclosure | Gmail App Password / Neon URL | mitigate | Secrets live only in Vercel env (encrypted) and the gitignored `.env.vercel.local`; never committed, never logged. The human sets SMTP_PASS in the Vercel dashboard (holds the secret); `.env.example` carries placeholders only. |
| T-04-06 | Spoofing | prod EMAIL_FROM (D-03 DMARC trap) | mitigate | Prod uses `smtp.gmail.com` with `EMAIL_FROM = SMTP_USER` (self-aligns SPF/DKIM/DMARC); a gmail From is NEVER used on a non-gmail relay. Documented in the checkpoint + `.env.example`. |
| T-04-11 | Tampering | Neon migration (schema drift) | mitigate | Apply the SAME reviewed additive `0002_*.sql` from 04-02 via `db:migrate` (non-interactive); verify `winning_option_id` on Neon before relying on it. No existing column altered. The migrate runs BEFORE the deploy so prod code never hits a missing column. |
| T-04-13 | Denial of Service | Gmail 100/day automated-SMTP cap | accept | Volume (~5–30/poll, a few polls/month) is far under any plausible cap; the `rateLimited` UI (04-01) surfaces a 429 visibly and degrades to copy-link if ever hit. SMTP2GO single-sender is the pre-wired fallback behind the same D-01 seam. |
