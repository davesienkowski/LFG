# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4: Email & Finalization.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Continuity note:** Reuses the exact `base-nova / neutral / base-ui` preset and
> the Phase 1/2/3 spacing/typography/color tokens verbatim — this phase adds
> **zero new shadcn components** (per RESEARCH.md's "zero new runtime
> dependencies" directive) and **zero new design tokens**. The only genuinely
> new visual vocabulary is (1) a per-recipient send-result chip (reuses the
> exact icon+label chip mechanic Phase 2/3 already established for
> `STATE_META`, just with a send-status palette instead of a vote-state
> palette), (2) an inline two-step "Book it" confirm disclosure (reuses the
> amber "Keep private" caution-badge language from Phase 1 for the warning,
> and the emerald "Best"/`BestDayBadge` language from Phase 3 for both the
> pre-selected suggestion and the post-finalize "Booked" state), and (3) three
> plain-HTML email templates that translate this same token system into
> email-client-safe inline CSS (per D-10 — no `react-email`).

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | shadcn/ui | `components.json` (detected, unchanged since Phase 1) |
| Preset | `base-nova` style, `neutral` base color, CSS variables enabled | `components.json` |
| Component library | `@base-ui/react` primitives (NOT Radix) | confirmed via `button.tsx`/`input.tsx` (unchanged since Phase 1) |

**No new shadcn components required for Phase 4.** Reuse `card`, `button`, `input`, `label`, `textarea` only. The "Book it" confirm step is a **hand-written inline disclosure** (a second Card region that appears in place, not a modal) — `@base-ui/react` ships a `Dialog`/`AlertDialog` primitive that shadcn could wrap, but per Phase 3's established precedent ("prefer a hand-written composition over installing a new component for a single-use, low-complexity interaction") this phase does the same: no `alert-dialog` install, no focus-trap/portal complexity, fully consistent with the existing "calm, minimal" visual language and zero new dependencies.

| Component | Used On | Usage |
|-----------|---------|-------|
| `card` | Invite-by-email card, Book-it card, Poll-finalized card, unconfigured-email notice | Reused verbatim, `p-6` interior per existing pattern |

| `input` | Not used this phase | Multi-address entry uses `textarea`, not `input` — addresses are naturally multi-line/comma-separated (see Component Spec) |
| `label` | Invite-by-email card | "Email addresses" label above the textarea |
| `textarea` | Invite-by-email card | Multi-address free-text entry |
| `<span>` pill (hand-styled) | Send-result chips, "Booked" badge | Same non-installed-`Badge` pattern already used for "Keep private" (Phase 1) and `BestDayBadge` (Phase 3) |

## Spacing Scale

Reused verbatim from Phase 1/2/3 — no new tokens. All values are multiples of 4.

| Token | Value | Tailwind class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `gap-1` / `p-1` | Icon-to-label gap inside a send-result chip |
| sm | 8px | `gap-2` / `p-2` | Chip internal gap, textarea-to-button gap |
| md | 16px | `gap-4` / `p-4` | Section gap (heading → control → result list) |
| lg | 24px | `gap-6` / `p-6` | Card interior padding (unchanged Card default), section padding |
| xl | 32px | `gap-8` / `p-8` | Not used this phase |
| 2xl | 48px | `gap-12` / `p-12` | Page-level vertical padding (inherited `<main>` container, unchanged) |

**Exceptions (new for Phase 4):**

- Send/Book buttons: minimum height **44px** (`h-11`, or `size="lg"` + explicit `h-11` override) — same touch-target rule Phase 1/2/3 apply to every interactive control on this project. The shadcn `button` default height (`h-8`) is below 44px and MUST be overridden on every primary action button introduced this phase, exactly as Phase 3 already overrides the native `<select>`/"Clear filter" button.
- Email templates: a single fixed **600px** max content width (`max-width: 600px`) — the one email-specific spacing exception, driven by email-client rendering norms rather than the app's 8-point scale; the *internal* padding within that 600px column still uses the 8-point scale (`24px` outer padding, `16px` paragraph gaps) so the two systems stay visually consistent when a recipient views the email next to the app.

## Typography

**Reused verbatim from Phase 1/2/3 — same 5 roles, same 2 weights. No new sizes or weights introduced.**

| Role | Size | Weight | Line Height | Tailwind | Usage this phase |
|------|------|--------|-------------|---------|-------|
| Display | 30px | 600 (semibold) | 1.1 | `text-3xl font-semibold leading-tight` | Unchanged — poll title, above this phase's additions |
| Heading | 24px | 600 (semibold) | 1.2 | `text-2xl font-semibold leading-snug` | "Invite by email", "Book it", "Poll finalized" section headings; confirm-step heading; closed/read-only banner heading |
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-relaxed` | Card helper copy, confirm-step warning text, empty-state body |
| Label | 14px | 600 (semibold) | 1.5 | `text-sm font-semibold leading-normal` | "Email addresses" field label, per-recipient email address text |
| Caption / Badge | 12px | 400 or 600 (regular or semibold) | 1.5 | `text-xs font-normal` / `text-xs font-semibold` | Send-result chip text, "Booked" badge, "Suggested best day" badge, rate-limit caption |

**Caption / Badge role — approved 5th typography role, reused verbatim (no new size/weight introduced by Phase 4).** This is not a new size added by this phase. `text-xs` (12px) is the project's formally-approved 5th typography role: it was declared and justified in the SHIPPED Phase 3 contract (`.planning/phases/03-results-dashboard/03-UI-SPEC.md:78-80`), which itself traces the role back to Phase 1's "Keep private" badge (`src/app/a/[adminUrlId]/page.tsx` — `text-xs font-semibold`), already in production. Phase 3's contract formally declared 12px as the project's 5th typography role and it passed the Phase 3 UI checker under that declaration. Phase 4 reuses this same role verbatim for its own secondary/caption/badge text (send-result chip text, "Booked"/"Suggested best day" badges, rate-limit caption) — zero new sizes and zero new weights are introduced by this phase; the weight budget stays at 2 (regular 400 + semibold 600), matching the Phase 3 precedent exactly.

Font stack (app UI): `Inter, system-ui, -apple-system, sans-serif` (unchanged).

**Email HTML uses its own stack (new, scoped to `templates.ts` only):** `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif` — `next/font`'s Inter is not usable inside emails (most clients strip `<link>`/`@font-face` webfont loading), so templates fall back to the standard email-safe system stack. Sizes inside emails are expressed in `px` inline styles mapped 1:1 to the same 4 roles: Heading 24px/600, Body 16px/400 (line-height 1.5 via inline `line-height:24px`), Label 14px/600, Caption 12px/400 — the numeric values are identical to the app; only the delivery mechanism (inline `style=""` instead of Tailwind classes) differs.

## Component Spec: Invite-by-email Card (MAIL-01/02/03, D-05/06)

Mounted as a third Card inside the existing "Share your poll" section, after the Participant-link and Admin-link Cards (page order unchanged above this addition — see Layout Skeleton).

### Structure (email configured)

```tsx
<Card className="flex flex-col gap-4 p-6">
  <div className="flex flex-col gap-1">
    <h3 className="text-2xl font-semibold leading-snug">Invite by email</h3>
    <p className="text-base text-muted-foreground">
      Enter one or more email addresses (comma or newline separated).
      Each person gets their own invite — no one sees anyone else's address.
    </p>
  </div>

  <div className="flex flex-col gap-1">
    <Label htmlFor="invite-addresses">Email addresses</Label>
    <Textarea
      id="invite-addresses"
      name="addresses"
      placeholder="alex@example.com, sam@example.com"
      className="min-h-24"
      disabled={isPending}
    />
  </div>

  <div>
    <Button type="submit" className="h-11" disabled={isPending}>
      {isPending ? "Sending…" : "Send invites"}
    </Button>
  </div>

  {/* Per-recipient results — rendered after a submit, persists until next submit */}
  {results.length > 0 && (
    <ul className="flex flex-col gap-2" aria-live="polite">
      {results.map((r) => (
        <li key={r.email} className="flex items-center gap-2 text-sm">
          <span className="font-semibold">{r.email}</span>
          <span className={cn(
            "inline-flex items-center gap-1 rounded-full border px-2 py-0.5 text-xs font-semibold",
            SEND_STATUS_META[r.status].className,
          )}>
            <Icon aria-hidden className="size-3.5" />
            {SEND_STATUS_META[r.status].label}
          </span>
        </li>
      ))}
    </ul>
  )}
</Card>
```

- Address parsing (comma AND newline separated) happens before `sendEmail()` is ever called; each address is Zod-validated (`z.string().email()`) — a malformed entry renders its own `Failed` chip with a validation-specific message ("Not a valid email address") rather than being silently dropped or aborting the batch.
- Sends run in the synchronous per-recipient loop from RESEARCH.md Pattern 3 — the result list only appears after the action completes (no per-address streaming/progressive reveal this phase; matches D-06's "synchronous inside the server action").
- `aria-live="polite"` on the result list so screen-reader users hear the outcome once, without needing to navigate back to the top of the form.
- Textarea and button share the disabled state (`isPending`) — no partial-submit state.

### Structure (email NOT configured — MAIL-03)

```tsx
<Card className="flex flex-col gap-2 p-6">
  <h3 className="text-2xl font-semibold leading-snug">Email isn't set up</h3>
  <p className="text-base text-muted-foreground">
    Copy the participant link above and share it manually.
  </p>
</Card>
```

- No textarea, no button, no second `CopyLinkButton` — this card is purely explanatory; the actionable copy button already lives in the Participant-link Card directly above it (D-05: "degrades to the existing `CopyLinkButton`").
- This is a first-class, expected state (`EMAIL_PROVIDER` unset/`"none"`) — never an error banner, never styled with destructive/amber.

## Component Spec: Book-it Finalize Control (FNL-01/02/03, D-08/09)

Mounted as its own section below "Results" (see Layout Skeleton) — the organizer reviews the grid, then decides. Two states: **picker** (poll open) and **finalized** (poll closed). Only one renders at a time.

### Picker state (poll open)

```tsx
<div className="flex flex-col gap-4">
  <h2 className="text-2xl font-semibold leading-snug">Book it</h2>
  <p className="text-base text-muted-foreground">
    Pick the date you're going with. This closes voting for everyone.
  </p>

  <Card className="flex flex-col gap-3 p-6">
    <fieldset className="flex flex-col gap-2">
      <legend className="text-sm font-semibold">Candidate dates</legend>
      {options.map((opt) => (
        <label key={opt.id} className="flex items-center gap-2 text-base">
          <input
            type="radio"
            name="winningOptionId"
            value={opt.id}
            defaultChecked={opt.isBest}
            className="size-4"
          />
          {formatDateWithTime(opt.date, opt.startTime?.slice(0, 5) ?? null)}
          {opt.isBest && (
            <span className="inline-flex items-center rounded-full border border-emerald-300 bg-emerald-100 px-2 py-0.5 text-xs font-semibold text-emerald-800">
              Suggested
            </span>
          )}
        </label>
      ))}
    </fieldset>

    {!showConfirm ? (
      <div>
        <Button type="button" className="h-11" onClick={() => setShowConfirm(true)}>
          Book this date
        </Button>
      </div>
    ) : (
      <div className="rounded-lg border border-amber-200 bg-amber-50 p-4 flex flex-col gap-3">
        <p className="text-base">
          This closes voting for everyone. Participants who already voted will
          get an email with this date. This can't be undone.
        </p>
        <div className="flex gap-2">
          <Button type="submit" className="h-11" disabled={isPending}>
            {isPending ? "Booking…" : "Confirm and close poll"}
          </Button>
          <Button
            type="button"
            variant="ghost"
            className="h-11"
            onClick={() => setShowConfirm(false)}
            disabled={isPending}
          >
            Keep poll open
          </Button>
        </div>
      </div>
    )}
  </Card>
</div>
```

- Radio group pre-selects the `isBest` option(s) from `computeResults` (D-08) — if multiple dates tie for best, pre-select the chronologically-first tying option (native radio groups permit exactly one `defaultChecked`); the organizer may still pick any other candidate date.
- "Book this date" is disabled only in the trivial case of zero candidate dates (never possible in this app — `POLL-03` requires at least one date at creation) — otherwise always enabled once a radio is selected (one is always pre-selected, so the button is enabled on first render).
- Clicking "Book this date" **does not submit anything yet** — it only reveals the amber confirm panel (`showConfirm` is local client state, not a server round-trip). The actual `closePoll` server action only fires on "Confirm and close poll".
- "Keep poll open" collapses the confirm panel back to the plain picker with no side effects — matches the reversible, no-cost-to-cancel precedent already used for Phase 3's "Clear filter".
- Native `<input type="radio">` (not a shadcn `RadioGroup` — not installed, and a plain radio group needs no extra dependency), consistent with Phase 1/3's "native form control over a new shadcn install" precedent.

### Finalized state (poll closed)

```tsx
<Card className="flex flex-col gap-2 p-6 border-emerald-200 bg-emerald-50/40">
  <h2 className="text-2xl font-semibold leading-snug">Poll finalized</h2>
  <p className="text-base text-muted-foreground">
    {formatDateWithTime(winningOption.date, winningOption.startTime)} is
    booked. Everyone who voted and gave an email should get a confirmation.
  </p>
</Card>
```

**Copy note (prohibition-probe finding, see Prohibitions section below):** the body text says "should get a confirmation," not "was notified" — finalization emails are sent best-effort (D-09) and per-recipient failures are never surfaced back to this Card, so asserting unconditional past-tense delivery ("was notified") would overstate a guarantee the system cannot actually back. This is a deliberate, probed wording choice, not a typo.

- Replaces the entire picker — never rendered alongside it.
- No re-open/undo control anywhere on the page (v2 deferred, D-08) — this Card is purely informational.
- The "Booked" badge (see Color section) additionally renders next to `<h1>{poll.title}</h1>` at the top of the page — the organizer sees the outcome immediately on page load, before scrolling to this Card.

## Component Spec: Email Templates (VOTE-04 / MAIL-01 / FNL-03, D-10)

Three plain-HTML templates in `src/lib/email/templates.ts`. **No `react-email`.** Each shares one shell so the visual language stays consistent across all three sends; only the heading, body copy, and CTA differ.

### Shared shell (email-client-safe)

```html
<!-- Outer table: bulletproof centering/width for Outlook + all major clients.
     Inner content is plain inline-styled div/p/a — safe because it's a single
     column inside one <td>, not a multi-column layout. -->
<table role="presentation" width="100%" cellpadding="0" cellspacing="0" style="background-color:#ffffff;">
  <tr>
    <td align="center" style="padding:24px 16px;">
      <div style="max-width:600px; width:100%; font-family:-apple-system,BlinkMacSystemFont,'Segoe UI',Roboto,Helvetica,Arial,sans-serif; color:#171717;">
        <!-- Heading: 24px/600, matches app Heading role -->
        <p style="font-size:24px; font-weight:600; line-height:1.2; margin:0 0 16px 0;">{heading}</p>
        <!-- Body: 16px/400, line-height 1.5 -->
        <p style="font-size:16px; font-weight:400; line-height:1.5; margin:0 0 24px 0; color:#171717;">{bodyText}</p>
        <!-- Event-details block (finalization email only) -->
        {eventDetailsBlock}
        <!-- Bulletproof CTA button -->
        <a href="{ctaUrl}" style="display:inline-block; background-color:#171717; color:#ffffff; font-size:16px; font-weight:600; text-decoration:none; padding:12px 20px; border-radius:8px; margin:0 0 16px 0;">{ctaLabel}</a>
        <!-- Fallback plain-text link, always present in case the button doesn't render/is stripped -->
        <p style="font-size:14px; font-weight:400; line-height:1.5; color:#737373; margin:0;">
          Or copy this link: <a href="{ctaUrl}" style="color:#171717;">{ctaUrl}</a>
        </p>
      </div>
    </td>
  </tr>
</table>
```

- Hex approximations of the app's OKLCH tokens (email clients cannot render `oklch()`): `#171717` (≈ `--foreground`/`--primary` near-black), `#ffffff` (≈ `--background` white), `#737373` (≈ `--muted-foreground`).
- Event-details block (finalization email only) reuses the same near-white card language as the app's Card component, translated to inline styles:
  ```html
  <div style="border:1px solid #e5e5e5; background-color:#fafafa; border-radius:8px; padding:16px; margin:0 0 24px 0;">
    <p style="font-size:14px; font-weight:600; margin:0 0 4px 0;">{poll.title}</p>
    <p style="font-size:16px; font-weight:400; margin:0 0 4px 0;">{formatDateWithTime(chosenDate, chosenTime)}</p>
    {location && <p style="font-size:14px; color:#737373; margin:0;">{location}</p>}
  </div>
  ```
- Dates render **only** via the existing `formatDateWithTime` (never `new Date()` on the date-only string) — identical rule as every other surface in this app (D-11/P3, carried into email templates).
- No images, no external CSS/webfont `<link>`, no `<script>` — every style is inline, matching the "table-free-or-simple" email-client-safety directive.

## Interaction States

### Invite — Send

- Clicking "Send invites" disables the textarea + button, shows "Sending…" (no separate spinner icon — text-only pending state, matches the existing `vote-form.tsx`/`copy-link-button.tsx` pending-label convention).
- On completion, the per-recipient result list renders below the button — one row per parsed address, each an independent chip (Sent / Rate limited / Failed). One failure never blocks or hides the others (D-05).
- Re-submitting (e.g., after fixing a failed address) replaces the previous result list entirely — no accumulation across multiple submits within the same page view.
- `aria-live="polite"` announces the result list once it renders.

### Invite — Not Configured (MAIL-03)

- The empty-state Card renders on every page load when `EMAIL_PROVIDER` is unset/`"none"` — this is a build-time/env-derived state, not a client-side toggle; there is no button to "enable" it from the UI.

### Book It — Picker → Confirm → Keep poll open

- Selecting a different radio button changes which date the eventual `closePoll` submit will use — purely client state, no network call (matches Phase 3's filter-control precedent of zero round-trips for non-committing interactions).
- Clicking "Book this date" reveals the amber confirm panel in place (no page navigation, no modal/portal) — the radio group remains visible and still changeable while the confirm panel is open (changing the selection while the panel is open keeps the panel open, reflecting the newly-selected date's copy — the confirm copy is date-agnostic, so no re-render of the confirm text is needed).
- Clicking "Keep poll open" collapses the confirm panel with zero side effects — equivalent to the panel never having been opened.
- Clicking "Confirm and close poll" submits the actual `closePoll` server action; the button enters a pending state ("Booking…") and both buttons disable during the pending window (matches the disabled-both-during-pending pattern already used in `vote-form.tsx`).
- On success, the page re-renders (via the existing RSC + redirect/revalidate pattern already used elsewhere in this app) into the **Finalized state** — the picker Card is completely replaced by the "Poll finalized" Card; there is no transient "just booked!" toast — the finalized Card itself IS the confirmation, consistent with this app's no-toast precedent (Phase 1/2 confirmations are always a persistent on-page state, e.g., `/thanks`, never an ephemeral toast).

### Book It — Post-Finalize (Read-Only)

- Once closed, there is no path back to the picker anywhere in the UI (no "reopen" button exists per D-08 — v2 deferred). This is intentional, not an omission.
- The existing Phase 2 "Voting is closed" banner on the participant/edit pages is unaffected by this phase's changes — reused verbatim, no restyle.

## Accessibility Requirements

- **Every send-result and finalize-state signal communicates via icon + text label, never color alone** — identical invariant to Phase 2/3's `STATE_META`/`BestDayBadge` rule, applied here to the Sent/Rate-limited/Failed chips and the "Suggested"/"Booked" badges.
- Textarea has an associated `<Label htmlFor>` (native semantics, unchanged shadcn `Label`/`Textarea` pairing already used elsewhere in this app).
- The radio group uses a native `<fieldset>`/`<legend>`/`<input type="radio">` — fully keyboard-operable (arrow keys move selection) and screen-reader-announced (`legend` gives the group a name) without any custom ARIA.
- Minimum touch target: 44px (`h-11`) for "Send invites", "Book this date", "Confirm and close poll", and "Keep poll open" — same override pattern already applied throughout Phase 1/2/3.
- `aria-live="polite"` region on the invite result list, mirroring Phase 2/3's live-region precedent for asynchronous, non-navigational state changes.
- The amber confirm panel is **not** a modal (no focus trap, no `aria-modal`) — it is inline content that receives normal document flow and tab order; this is an intentional accessibility simplification (no focus-trap bugs possible) consistent with the "hand-written disclosure over a Dialog" decision above.
- Email templates: every link (CTA button and fallback plain-text link) carries descriptive link text (never bare "click here") — "View the poll & vote", "View or edit my response", "View the poll" — screen readers and email clients that strip button styling both get meaningful link text.
- Chip/badge colors (`emerald-50/700`, `amber-50/700`, `destructive/10 + destructive`) are the exact palette already verified against WCAG AA text contrast (4.5:1) in Phase 2/3 — no new contrast verification needed for the app UI; email-HTML hex approximations (`#171717` on `#ffffff`, `#737373` on `#ffffff`) both clear 4.5:1 as well (near-black on white, and mid-gray `#737373` ≈ 4.6:1 on white — verify at implementation time with a contrast checker since email clients cannot inherit the app's already-verified CSS variables).
